# CHANGELOG

## v7.4.1-stabilization

Архитектурная стабилизация после рабочей версии `v7.4.0-stable`.

### Что перенесено в основную архитектуру

- Блок «Стороны сделки» перенесен из динамического патча в основной `index.html`.
- Блок «Финансы / комиссии / расходы» перенесен из динамического патча в основной `index.html`.
- `financeRestore.js` больше не создает поля формы. Теперь он отвечает только за вкладку «Финансы» и финансовую сводку.
- Блок «Стороны и деньги» в «Сводке» перенесен в `assets/js/ui/render.js`.
- Блок «Стороны / комиссии / расходы для юриста» перенесен в `assets/js/ui/render.js`.
- Карточка брокеру дополнена количеством покупателей, основным покупателем и расходами банка.
- `auditAndPatches.js` больше не добавляет рабочие бизнес-блоки в интерфейс. Теперь это диагностический модуль вкладки «Проверка».

### Что нужно проверить после обновления

1. Вкладка «Проверка» — пункты про сводку, юриста и финансы должны быть зелеными.
2. Вкладка «Тесты» → «Проверить все сценарии».
3. В «Сводке» должен быть блок «Стороны и деньги».
4. В «Юристу» должен быть блок «Стороны / комиссии / расходы для юриста».
5. В «Финансы» должны корректно отображаться комиссии и расходы.
6. Сохранение в Supabase должно сохранять поля сторон и финансов.
7. Открытие сделки из Supabase должно возвращать поля сторон и финансов.

### Оставшиеся архитектурные долги

- `cloudAutoPatch.js` пока остается мостом между основной страницей и открытой Supabase-сделкой.
- `loginFix.js` пока остается усиленным модулем авторизации и диагностики входа.
- Нужна отдельная страница `deals.html` для полноценного списка сделок.
- Нужны статусы сделки как отдельный рабочий процесс.
- Нужно связать решения юриста/брокера/менеджера с автоматическим созданием задач.
- Нужен экспорт/печать PDF для юриста, брокера и клиента.

## v7.4.0-stable

Стабильная рабочая точка перед дальнейшим развитием мини-CRM.

### Что работает

- Основной навигатор сделки СПН.
- Быстрые сценарии и ручное заполнение.
- Сводка по сделке.
- Карточка юристу.
- Документы продавца, покупателя и банка.
- Сообщения клиенту.
- Локальная специфика Борисоглебска.
- Восстановленные блоки сторон сделки.
- Восстановленные блоки финансов, комиссий и расходов.
- Вкладка «Финансы».
- Вход в Supabase без зависимости от esm.sh.
- Понятный статус авторизации.
- Сохранение сделки в Supabase.
- Открытие сделки из Supabase.
- Обновление открытой сделки.
- Вкладка «Решения» для юриста, брокера, менеджера и админа.
- Вкладка «Задачи» по сделке.
- Вкладка «Проверка».
- Вкладка «Тесты».

### Техническое состояние

- GitHub Pages работает из корня репозитория.
- Supabase используется через отдельные таблицы с префиксом `nav_`.
- Основной проект в Supabase не должен конфликтовать с навигатором.
- Вход и REST-запросы работают через собственный минимальный Supabase REST/Auth клиент.
- Добавлены тесты для типовых сценариев: квартира, Сбер + маткапитал, дом + участок, доля, несовершеннолетний собственник, НИС.

### Исторические архитектурные долги версии v7.4.0

- Часть функциональности была подключена как динамические патчи: `financeRestore.js`, `auditAndPatches.js`, `cloudAutoPatch.js`, `loginFix.js`.
- Финансы и стороны требовалось перенести из патчей в основную форму.
- Финансовые блоки карточки юристу требовалось перенести из патчей в `ui/render.js`.

### Перед доработками обязательно проверить

1. Вкладка «Проверка» — все основные блоки должны быть зелеными.
2. Вкладка «Тесты» → «Проверить интерфейс».
3. Вкладка «Тесты» → «Проверить все сценарии».
4. Вход в Supabase.
5. Сохранение тестовой сделки.
6. Открытие сохраненной сделки.
7. Повторное сохранение открытой сделки.
8. Вкладка «Решения».
9. Вкладка «Задачи».
