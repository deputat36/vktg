# Navigator v2 — полный аудит и автономный план

Дата: 2026-07-13  
Базовая версия: `main` после PR #238, commit `c0cb0c2d5f36ee5e1a665fa6199f90911b73ac61`.

## 1. Общий вывод

Navigator v2 уже является сильной системой диагностики и подготовки сделки. Он собирает сценарий, выявляет юридические и операционные риски, формирует документы и задачи, рассчитывает готовность, распределяет доступ и даёт рабочие пространства основным ролям.

После PR #224–#238 закрыты многие пробелы предыдущего аудита:

- role-aware стартовый экран;
- правдивая операционная готовность и менеджерская очередь;
- task taxonomy и nullable persisted task type/SLA preview;
- отдельный брокерский кабинет;
- viewer workspace;
- lawyer focus mode;
- понятный handoff после сохранения СПН;
- разделение рабочей, административной и технической навигации;
- release drift report;
- восстановление фактической migration history для task-contract preview.

Главный незакрытый разрыв теперь не в количестве экранов. Система ещё не доказала на реальном пилоте полный цикл:

`сигнал → ответственный → срок → действие → подтверждение результата → следующий этап → контроль менеджера`.

Продукт пригоден для контролируемого пилота, но пока не готов к массовому запуску на весь отдел без isolated authenticated E2E, подтверждённого lifecycle задач и очистки операционных данных.

## 2. Оценка как разработчик

| Область | Оценка | Вывод |
|---|---:|---|
| Архитектура доступа | 8/10 | RPC-first, RLS и role guards реализованы последовательно |
| Бизнес-логика | 8.5/10 | Хорошо отражает реальные сценарии недвижимости |
| Безопасность | 7/10 | Anon RPC закрыты, но остаются Advisor/Auth/CSP вопросы |
| Тестируемость | 7/10 | Много focused regression checks, authenticated matrix заблокирован |
| Frontend-поддерживаемость | 5.5/10 | Много patch-модулей, ручных версий и высокой связности |
| Релизный процесс | 6.5/10 | Drift gate есть, автоматический controlled deploy ещё не завершён |
| Наблюдаемость | 7/10 | Health RPC и отчёты есть, нужен единый post-release verdict |
| Качество рабочих данных | 2/10 | Главный фактический риск пилота |

### Сильные стороны

- Browser работает через curated RPC layer, прямой доступ к Navigator-таблицам запрещён проверками.
- RLS и серверные проверки ролей находятся ближе к данным, а не только в интерфейсе.
- Helpers вынесены в private schema и имеют отдельный lockdown health.
- Edge Functions Navigator используют JWT verification.
- Рискованные изменения проверяются через read-only preview либо transaction/rollback rehearsal.
- Есть focused regressions для readiness, risk lifecycle, task taxonomy, broker queue, viewer, lawyer, SPN handoff и role routes.
- PR #235 ввёл read-only release drift baseline.
- PR #238 восстановил соответствие repository history фактической live migration `20260713151053`.

### Основные технические риски

1. Нет isolated Supabase target и disposable пользователей для реального authenticated browser matrix.
2. Invite, recovery и password flow не подтверждены end-to-end.
3. Shared Supabase project смешивает Navigator, legacy и другие подсистемы в Advisor.
4. 46 curated public RPC ожидаемо являются `SECURITY DEFINER` и требуют контролируемого whitelist, а не ручного запоминания.
5. Leaked-password protection выключена до проверки auth flow.
6. CSP всё ещё разрешает `unsafe-inline`.
7. Session и browser draft используют web storage.
8. Deal card и wizard остаются фрагментированными и высокосвязанными.
9. Build/cache-bust metadata частично обслуживаются вручную.
10. Нет единого TypeScript-контракта, сгенерированного из Supabase schema.
11. Release workflow выявляет drift, но не завершает approved deploy и post-deploy verification автоматически.
12. В общей migration history возможны расхождения версий при ручном применении; PR #238 исправил один фактический случай, но нужен постоянный gate.

## 3. Оценка как пользователь

### СПН

Что уже хорошо:

- понятный адаптивный мастер;
- сценарии близки к реальной работе;
- есть сохранение черновика;
- автоматически создаются юридические и брокерские сигналы;
- после сохранения показывается, кому передана сделка и что делать дальше.

Что мешает:

- маршрут остаётся длинным;
- комментарий иногда формально заменяет структурированное поле;
- обязательный минимум по сторонам и контактам недостаточно строгий;
- нет доказанного восстановления доступа/пароля;
- нужен ежедневный экран «что по моим сделкам требуется сегодня».

### Юрист

Что уже хорошо:

- отдельная приоритетная очередь;
- видны стоп-факторы, документы, возвраты и СПН;
- focus mode помогает начать со следующей важной сделки;
- карточка ведёт прямо к нужному разделу.

Что мешает:

- низкое качество входных данных увеличивает время проверки;
- подтверждение результата не всегда автоматически формирует следующий операционный шаг;
- authenticated desktop/mobile сценарий не подтверждён.

### Менеджер

Что уже хорошо:

- operational readiness не маскирует проблемы процентом заполнения анкеты;
- очередь показывает причину, владельца, срок и главное действие;
- видна нагрузка СПН;
- role-aware старт ведёт в контроль, а не в техническую диагностику.

Что мешает:

- часть действий требует перехода в карточку;
- нет самостоятельного события «контроль выполнен»;
- нужна группировка «сегодня / неделя / без срока / просрочено»;
- реальные сделки фактически не распределены по менеджерам.

### Брокер

Что уже хорошо:

- есть отдельная очередь и понятный маршрут.

Что мешает:

- отсутствует полный persisted lifecycle заявки: банк, сумма, статус, срок решения, причина отказа;
- результат финансирования не всегда создаёт следующий шаг СПН и менеджеру.

### Viewer

Что уже хорошо:

- есть отдельный read-only workspace;
- роль не выглядит как урезанный администратор.

Что мешает:

- реальный browser read-only E2E не выполнен;
- требуется authenticated visual audit на мобильном экране.

### Owner/Admin

Что уже хорошо:

- рабочая, административная и техническая навигация разделены;
- доступны команда, роли, доступы, качество данных и диагностика;
- release drift стал явным контролем.

Что мешает:

- опасные изменения профилей требуют более сильного preview/confirmation;
- нет компактного executive summary пилота;
- предупреждения shared-project Advisor до текущего среза не имели Navigator-only автоматического gate.

## 4. Live-состояние и риск внедрения

Последний подтверждённый read-only baseline:

- 21 сделка, из них 16 рабочих;
- все 16 требуют внимания менеджера;
- у всех 16 отсутствует менеджер или явное исключение;
- 2 сделки без СПН;
- 11 сделок ждут распределения юриста;
- 5 требуют брокера;
- у всех 16 есть блокирующий риск и просроченная задача;
- у 12 просрочен обязательный документ;
- 92 задачи существуют в таблице;
- persisted `task_type` = 0 и persisted `sla_days` = 0: существующие строки не переклассифицированы автоматически.

Пользовательский риск: система может выглядеть как бесконечная красная очередь, если команда не видит, что выполнение действий действительно уменьшает backlog и переводит сделки дальше.

## 5. Приоритетный автономный план

### P0 — доказать безопасный вход и реальный пилот

1. Создать isolated Supabase branch/project и disposable role accounts.
2. Запустить authenticated matrix для owner/admin/manager/spn/lawyer/broker/viewer.
3. Проверить invite/recovery/password flow.
4. Выбрать 2–3 реальные сделки для пилота.
5. Назначить менеджеров и СПН только после подтверждения правильных людей.
6. Провести одну сделку через полный lifecycle до подтверждённого результата.

Блокеры, которые нельзя обходить автономно:

- создание платной Supabase branch требует cost confirmation;
- реальные назначения требуют бизнес-подтверждения владельца;
- disposable mailbox и credentials должны быть настроены отдельно от production users.

### P1 — замкнуть ежедневную операционную работу

1. Read-only operational adoption report: активность, реальные действия, закрытые задачи/риски/документы, сделки без движения.
2. Audited point mutation для task contract: preview → одна задача → audit event → reload verification.
3. Действия менеджера из очереди с preview и confirmation.
4. Событие подтверждения результата контроля.
5. Автоматическое формирование следующего шага после закрытия задачи, риска или документа.
6. Полный broker lifecycle и обратный handoff.
7. Группировка очередей по срокам.

### P2 — безопасность и качество

1. Navigator-only Advisor scope и CI gate.
2. После auth E2E включить leaked-password protection и повторить invite/recovery/password tests.
3. Поэтапно усилить CSP без поломки рабочих страниц.
4. Исследовать более безопасное хранение session/draft.
5. Все admin/role mutations проводить через preview, confirmation и audit event.
6. Не трогать unrelated subsystems и legacy без отдельного решения.

### P3 — поддерживаемость и релиз

1. Единый автоматический build manifest и cache-bust.
2. TypeScript types из Supabase schema.
3. Снижать связность deal card и wizard только по функциональным границам.
4. Перевести CSS к feature-файлам и дизайн-токенам.
5. Добавить contract tests для payload и state transitions.
6. Завершить controlled release lifecycle: approval → migration/Edge deploy → smoke → post-deploy verdict → baseline update.
7. Усилить migration-history gate, чтобы live version всегда имела точный source file в repository.

## 6. Правила автономной работы

- Не менять реальные сделки массово.
- Не назначать людей по догадке.
- Не закрывать backlog фиктивными статусами.
- Не выполнять production DDL вне migration и controlled release.
- Не отзывать RPC grants только ради зелёного Advisor.
- Не удалять индексы только по сигналу unused index.
- Не трогать `leader_*`, `parket_*`, `broker_*` и legacy `nav_*` в Navigator PR.
- Каждый срез оформлять небольшим PR с focused regression и проверяемым rollback/preview.
- После merge сверять live state и обновлять handoff фактическими результатами.

## 7. Уже выполнено в рамках этого запуска

- в repository добавлена точная historical copy под live version;

- предупреждения других подсистем не выдаются за исправленные.

## 8. Следующий автономный срез

После merge Advisor gate следующий безопасный приоритет — read-only operational adoption report без изменения рабочих строк. Он должен ответить владельцу и менеджеру:

- какие сделки реально двигались за 7/14/30 дней;
- какие действия закрывались, а какие только создавались;
- сколько задач, рисков и документов подтверждённо завершено;
- где нет владельца, срока или следующего шага;
- какие сделки создают повторяющиеся quality warnings вместо клиентских действий.

Только после этого можно проектировать точечные mutation controls и пилотировать их на одной подтверждённой сделке.
