# Navigator v2 — privacy-safe UX measurement contract

## Статус

Контракт работает в режиме `event-only`.

- браузер не отправляет события по сети;
- события не сохраняются в `localStorage`, `sessionStorage`, IndexedDB, cookies или Supabase;
- новые RPC, таблицы, migrations, Edge Functions и grants не требуются;
- обработчик по умолчанию только создаёт локальный `CustomEvent` `nav-v2:ux-measurement`;
- если слушателя нет, событие сразу исчезает и нигде не накапливается.

Цель этапа — определить безопасный формат измерений и проверить его в браузере до принятия решения о backend-агрегации.

## Что измеряет браузер

### `primary_action_opened`

Пользователь открыл главное действие текущего рабочего экрана.

Разрешённые измерения:

- экран: `dashboard`, `deals`, `deal_card`, `manager`;
- viewport: `compact`, `mobile`, `desktop`;
- тип действия из закрытого справочника;
- позиция: `primary` или `secondary`;
- диапазон времени от загрузки измерительного модуля до клика.

Локальный клик не считается выполненной задачей, проверенным документом, устранённым риском или завершённой сделкой.

### `secondary_details_opened`

Пользователь впервые раскрыл дополнительный контекст на текущей странице.

Событие показывает потребность в дополнительных данных, но не является ошибкой, завершением работы или оценкой сотрудника.

## Что определяется только по серверному payload

Pure-модель `ux-server-measurement-model-v2.js` формирует безопасные кандидаты измерений из уже загруженных `events`, `tasks`, `documents`, `risks` и текущего состояния сделки.

Она не подключена к browser transport и ничего не отправляет.

### `server_result_observed`

Фиксируется только когда существующая модель completion evidence подтверждает совпадение audit event с текущим серверным состоянием.

Допустимые типы результата:

- `task`;
- `document`;
- `risk`;
- `deal_status`.

Page view, кнопка, локальный toast или обещание интерфейса результатом не считаются.

### `spn_rework_return_observed`

Есть серверное событие возврата СПН на доработку.

### `spn_rework_submitted_observed`

После возврата найдено более позднее серверное событие повторной отправки. Передаётся только диапазон длительности `возврат → повторная отправка`.

### `spn_recheck_observed`

После повторной отправки найдено более позднее серверное решение проверки. Передаётся только диапазон длительности `повторная отправка → решение`.

## Схема события

Разрешены только поля:

- `schema_version`;
- `event_name`;
- `event_source`;
- `surface`;
- `viewport`;
- `action_kind`;
- `action_slot`;
- `result_type`;
- `duration_bucket`.

Все значения выбираются из закрытых enum-справочников. Неизвестные поля игнорируются.

## Запрещённые данные

В события нельзя включать:

- UUID сделки, задачи, документа, риска, события или пользователя;
- адрес объекта;
- ФИО клиента или сотрудника;
- телефоны и email;
- комментарии, причины возврата и другой свободный текст;
- стоимость, размер задатка и финансовые параметры сделки;
- точные timestamps серверных действий;
- содержимое browser drafts;
- URL с query-параметрами сделки.

Роль пользователя не передаётся браузером. Если позднее потребуется ролевой разрез, он должен определяться на доверенной серверной стороне по authenticated context.

## Метрики для управленческого решения

### 1. Время до главного действия

Определение: распределение `duration_bucket` среди событий `primary_action_opened`.

Использование: понять, помогает ли первый экран быстрее переходить к работе.

Ограничение: учитываются только пользователи, которые нажали действие. Метрика не является конверсией page view → click, пока не появится отдельно одобренный безопасный denominator.

### 2. Подтверждённые результаты

Определение: количество и структура `server_result_observed` по типам результата.

Источник истины: только серверное audit event + совпадающее текущее состояние сущности.

Ограничение: браузерная кнопка не увеличивает показатель.

### 3. Цикл доработки СПН

Определение:

- распределение времени `возврат → повторная отправка`;
- распределение времени `повторная отправка → решение проверки`.

Использование: искать системные задержки процесса, а не ранжировать отдельных сотрудников.

## Guardrails

- не строить персональные рейтинги сотрудников;
- не показывать разрезы с выборкой менее пяти завершённых циклов;
- не сравнивать роли без одинакового периода и сопоставимого типа работы;
- не считать UI-события доказательством завершения;
- не добавлять production write только ради удобства отчёта;
- не вводить retention до отдельного решения о целях, сроке хранения и доступах.

## Условия для будущего backend

Backend-агрегация допускается только отдельным изменением после доказанной необходимости.

До реализации должны быть утверждены:

1. конкретное управленческое решение, которое меняется от метрики;
2. серверная дедупликация событий;
3. минимальная выборка и период агрегации;
4. срок хранения;
5. роли с доступом;
6. запрет на свободный текст и идентификаторы;
7. pre/post privacy review;
8. отсутствие изменений рабочих строк сделки.

Текущий контракт не является разрешением на создание таблицы telemetry, RPC или production collector.
