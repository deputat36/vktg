# Navigator v2 — аудит создания сделки СПН

Дата среза: 17 июля 2026 года.

## 1. Граница аудита

Проверены актуальный `main`, `spn-v2.html`, основной мастер `spn-smart-v4.js`, все подключённые adaptive-модули, readiness/duplicate/idempotency guards, карточка сделки, очередь юриста, production-реализация `public.nav_v2_save_wizard_result(jsonb)` и сохранённые legacy-функции. Production Supabase исследован только read-only запросами; schema, RLS, grants, Auth, Edge Functions и рабочие строки не менялись.

Аудит оценивает продуктовый поток и системные контракты, а не работу сотрудников.

## 2. Подтверждённый baseline

## 3. Аудит по ролям

### Новый СПН

- Не знает заранее, почему простой выбор превращается в маршрут из 8–13 шагов.
- Встречает профессиональные формулировки до того, как зафиксирована сама ситуация.
- Вынужден либо пропустить вопрос, либо угадывать ответ; пропуск часто неотличим от «нет».
- Видит готовность в процентах, но не получает ясного ответа «можно ли просто сохранить» и «можно ли передать юристу».

### Опытный СПН

- Быстро понимает сценарий, но проходит повторяющиеся детали в отдельных блоках продавца, рисков и итоговой передачи.
- Не может создать карточку одним коротким проходом, а юридическую часть дозаполнить позже как отдельное действие.
- Может сохранить важное только в комментарии, который затем трудно фильтровать и проверять.

### Юрист

- Видит риски, документы и задачи, но должен восстанавливать смысл из нескольких вкладок и сырого snapshot.
- Не получает обязательную пару `тип запроса → ожидаемое решение`.
- Не видит в одном месте разделение фактов на подтверждённые документом, сообщённые клиентом и неизвестные.
- Общие автозадачи не всегда содержат владельца, срок, evidence и проверяемый результат.

### Менеджер

- Процент заполнения не объясняет, какой gate пройден и почему передача заблокирована.
- Нельзя уверенно отличить черновик, который СПН продолжает сам, от осознанной передачи профильной роли.
- Несогласованность frontend/backend затрудняет разбор маршрута после сохранения.

### Пользователь телефона

- Длинный меняющийся список шагов и несколько равнозначных действий увеличивают прокрутку и риск потери контекста.
- Adaptive DOM patches могут перестроить текущий экран после ответа.
- Визуальные блоки с большим числом вариантов превышают целевой бюджет 5–7 вариантов.

### Разработчик

- Бизнес-правила распределены между core, adaptive-модулями, readiness guard и SQL-функцией.
- Модули зависят от текста заголовков и порядка DOM, поэтому изменение формулировки может отключить patch без явной ошибки.
- Одни и те же признаки превращаются в риски, специалистов и handoff разными функциями.
- Старые сделки требуют совместимости, но единого адаптера старого snapshot к новому представлению нет.

### Специалист по безопасности

- Удаление полей имени и телефона adaptive-модулем правильно снижает сбор PII, но DOM-постобработка слабее единого исходного контракта.
- Адрес и деловые заметки допустимы только в минимальном объёме; паспортные данные, номера документов, реквизиты, подписи и сканы не нужны.
- Восстановление после неопределённого ответа сохранения раньше могло искать карточку только по типу объекта; это исправлено в PR #394.
- Новый прототип должен оставаться pure/repository-only до прохождения deployment gates.

## 4. Карта текущих вопросов

Колонка «до карточки» означает обязательность для быстрого черновика. «До handoff» — обязательность только перед осознанной передачей юристу.

| Текущий вопрос или блок | Зачем и кому | Системное влияние сейчас | Проблема | Новое решение | До карточки / до handoff | Влияние на юриста |
|---|---|---|---|---|---|---|
| Что готовим | Выбирает глубину сценария; СПН и система | Маршрут, readiness | Называется юридико-процессным режимом | Оставить бытовой вопрос «Что сейчас нужно?» с 5 вариантами | Да / да | `preparation_mode` и контекст запроса |
| Кого сопровождаем | Определяет стороны | Маршрут, документы | Затем отдельно спрашивается наличие продавца/покупателя | Оставить один источник истины; наличие стороны выводить из модели и стадии | Да / да | `representation_model`, side-aware документы |
| Стадия | Определяет срочность и блоки | Маршрут, задачи | Смешана с «нужен юрист» и четырьмя переключателями сторон | Оставить стадию; стороны и срочность вынести в отдельные короткие поля | Да / да | Объясняет первый шаг и срок |
| Продавец/покупатель уже есть | Управляет показом блоков | Маршрут | Дублирует сопровождение и может противоречить ему | Удалить как отдельный блок; показывать только при объективной неоднозначности | Нет / при необходимости | Меньше противоречий |
| Тип объекта | Выбирает объектный сценарий | Маршрут, риски, документы | Квартира выбирается в два шага; «не выбран» отсутствует как тип | Один список типов, включая «объект ещё не выбран» | Да / да | Объектный legal trigger |
| Адрес, кадастровый номер | Идентифицирует объект | Duplicate guard, карточка | Детали объекта показываются раньше, чем нужен быстрый черновик | Адрес/ориентир на этапе 1; кадастровый статус — короткий факт, номер не обязателен | Адрес или причина / объект обязателен | Быстрая идентификация без лишних данных |
| Подробности квартиры/комнаты/земли | Описывает объект | В основном snapshot и комментарии | Много свободного текста, слабая маршрутизация | Скрыть под «дополнительные сведения»; поднимать только факты, меняющие правила | Нет / только релевантное | Отдельные факты и документы вместо эссе |
| Признаки продавца | Выявляет детей, доли, супруга, доверенность | Риски, юрист, документы | Toggle означает только наличие; пропуск похож на «нет» | Четыре состояния + источник информации | Нет / релевантные факты | Подтверждённые, сообщённые и неизвестные факты |
| Основание права | Наследство, приватизация, суд | Риск и юрист | Список + комментарий, нет источника | Короткие вопросы только по сценарным основаниям; детали запрашивает юрист | Нет / да при риске | Конкретный request type и документы |
| Имя/телефон покупателя | Ранее использовались как минимальная заполненность | Readiness | Лишние персональные данные, не меняют legal route | Удалить; участника фиксировать ролью, не PII | Нет / нет | Никакого влияния; privacy улучшается |
| Источник денег | Ипотека, маткапитал, сертификат и др. | Брокер, юрист, риски | Frontend относит маткапитал/сертификат к брокеру | Раздельные факты; брокер только ипотека/военная ипотека | Нет / да при подготовке | Чёткое разделение брокера и юриста |
| Условия задатка | Сумма, дата, место, получатель | Readiness, snapshot | Несколько свободных полей до понимания допустимости задатка | Сначала три статуса: нужен ли, известна ли сумма, известны ли условия; детали позже | Нет / по типу запроса | Юрист понимает полноту без выдуманных значений |
| Расчёты | Определяет безопасную схему | Риск, готовность | Да/нет без «не знаю»; способы и согласованность смешаны | Отдельный статус согласования с четырьмя состояниями и раскрываемые известные условия | Нет / да для задатка/сделки | Явный блок и конкретный запрос |
| Расходы | Нужны до задатка | Риск, готовность | Повторяет проблему расчётов | Та же модель статуса; детали только если известны | Нет / да для задатка/сделки | Видно, что именно не согласовано |
| Дополнительные риски | Ловит остаточные сценарии | Риск, комментарий, задача | Повторяет продавца/объект, снова просит свободный текст | Удалить общий чек-лист; каталог сам показывает только активные факты | Нет / да для активных | Нет дублирования, прозрачное правило |
| Кто проверяет риск/срок/план | Пытается создать ответственность | Snapshot | Свободные поля без справочника и lifecycle | Генерировать bounded proposal: owner, action, deadline rule, evidence, expected result | Нет / после маршрута | Юрист получает конкретную работу |
| Ближайший шаг | Нужен СПН и менеджеру | Readiness, summary | Свободный текст почти всегда пуст в baseline | Предложить сценарные варианты, текст оставить дополнительным | Для карточки / да | Всегда виден следующий шаг СПН |
| Финальный комментарий СПН | Передаёт остаточный контекст | Handoff text | Пуст у всех 23 сделок baseline | Не делать несущим полем; дополнительный комментарий после структурированной сводки | Нет / нет | Не требуется искать смысл в комментарии |
| Вопрос юристу | Формулирует запрос | Частично adaptive snapshot | Полностью свободный и не обязателен | Автовыбор типа запроса + готовые варианты ожидаемого решения + optional comment | Нет / да | Главный заголовок юридической карточки |
| Процент готовности | Даёт общий сигнал | Save guard | Высокий процент скрывает критический пробел | Оставить вторичным, решения принимать по трём gates | Нет / нет | Причины блокировки видны явно |

## 5. Основные причины дефектов

1. Создание карточки и handoff профильной роли представлены одной финальной кнопкой.
2. Отсутствует каноническая модель `fact = value + evidence source`.
3. Правила дублируются в JS и SQL и не имеют общей версии.
4. Полезные данные сохраняются в snapshot, но не преобразуются в юридический паспорт.
5. Adaptive-слой исправляет интерфейс после рендера, а не строит его из одного route engine.
6. Документы и задачи выводятся из общего сценария, а не из `representation + trigger + owner + result`.

## 6. Решение по текущим модулям

| Компонент | Решение на этапе прототипа |
|---|---|
| `spn-smart-v4.js` | Не ломать production flow; использовать только как legacy источник и будущую точку интеграции. |
| Adaptive seller/buyer/money/deposit/terms/risks/finish | Не переносить DOM-patch архитектуру. Нужные смыслы перенести в versioned catalog и pure presentation model. |
| Draft, duplicate и idempotency guards | Сохранить поведение и покрыть совместимость отдельными сценариями. |
| Readiness guard | Не расширять процент; новый прототип использует gates, старый guard остаётся до интеграции. |
| `nav_v2_save_wizard_result` | Не менять в production на этапе прототипа. Подготовить будущую синхронизацию через тот же каталог и PG17 harness. |
| Deal card и lawyer queue | Добавить detached preview нового паспорта; старый payload показывать через legacy adapter. |

## 7. Приоритеты

P0: раздельные действия «сохранить» и «передать», explicit unknown, конкретный запрос юристу, mortgage-only broker, idempotency.

P1: единый каталог, юридический паспорт, gates, side-aware документы и конкретные task proposals.

P2: метрики интерфейса без PII и без оценки сотрудников; процент только как вспомогательная подсказка.
