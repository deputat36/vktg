# Navigator v2 design audit

Отдельный пункт релизной проверки PR #154. Его нельзя считать закрытым только по зеленым RPC/SQL-проверкам: дизайн проверяется вручную на реальных экранах и ролях.

## 1. Цель

Проверить, что Navigator v2 выглядит как единая рабочая система для офиса: без визуального шума, без разъезда сеток, без непонятных действий и без перегруза для ролей, которым диагностика не нужна.

## 2. Экраны для проверки

- `dashboard-v2.html` — рабочий стол.
- `deals-v2.html` — список сделок.
- `deal-card-v2.html` — карточка сделки.
- `spn-wizard-v2.html` — мастер СПН.
- `lawyer-v2.html` — очередь юриста.
- `admin-v2.html` — команда и диагностики.
- `nav-access-v2.html` — создание доступа сотруднику.
- `nav-accept-invite-v2.html` — принятие приглашения и установка пароля.

## 3. Роли для проверки

- owner/admin — видит диагностики, качество данных, команду и расширенные действия.
- manager — видит управленческие рабочие блоки без лишней системной диагностики.
- spn — видит свои сделки, свою зону ответственности и понятные действия по подготовке.
- lawyer — видит очередь и карточки без перегруза данными СПН, которые не нужны для юридической работы.
- broker/viewer — не видит лишних админских действий.

## 4. Обязательные дизайн-критерии

### 4.1 Единая визуальная система

- Одинаковые карточки, скругления, тени, отступы и плотность контента.
- Одинаковая логика заголовков: h1 для страницы, h2 для крупных блоков, h3 для внутренних секций.
- Кнопки одного уровня выглядят одинаково на всех страницах.
- Статусы и pills выглядят одинаково в списке, карточке и админке.

### 4.2 Адаптивность

Проверить ширины:

- 360 px — маленький телефон.
- 390-430 px — обычный телефон.
- 768 px — планшет.
- 1366 px — ноутбук.
- 1920 px — большой монитор.

На каждой ширине не должно быть:

- горизонтального скролла;
- наложения кнопок;
- обрезанных подписей;
- вылезающих email/ФИО/адресов;
- карточек, которые становятся слишком узкими для чтения.

### 4.3 Читаемость

- Длинные ФИО переносятся строками, а не ломают сетку.
- Email и UUID не раздвигают контейнеры.
- Адреса и комментарии переносятся внутри карточек.
- Малый текст остается читаемым на мобильной ширине.
- Важная информация не спрятана в слишком бледный цвет.

### 4.4 Цвета и состояния

- Зеленый — успешное состояние или активность.
- Желтый — предупреждение или неполные данные.
- Красный — ошибка, срочное действие или выключенный профиль.
- Синий — информационная метка.
- Опасные действия не выглядят как обычные вторичные кнопки.
- Основное действие на экране визуально заметнее остальных.

### 4.5 UX пустых состояний и ошибок

- Если данных нет, пользователь видит понятное объяснение, а не пустую область.
- Ошибка сессии ведет к повторному входу.
- Ошибка сохранения написана человеческим языком.
- Диагностические ошибки owner/admin не пугают рабочих ролей.

### 4.6 Ролевая чистота интерфейса

- СПН не видит лишний админский шум.
- Юрист видит, какие СПН ведут подготовку, но не получает перегруженный технический экран.
- Owner/admin видит диагностики и качество данных, но эти блоки отделены от ежедневной работы.
- Viewer не видит кнопок изменения данных.

## 5. Что считать дизайн-блокером

- Горизонтальный скролл на мобильной ширине.
- Наложение кнопок или карточек.
- Невозможность понять главное действие страницы.
- Админские действия видны роли, которой они не должны быть доступны.
- Красные/желтые предупреждения выглядят одинаково и не помогают принять решение.
- Ошибка отображается сырым техническим текстом там, где пользователь должен действовать.

## 6. Что считать некритичным улучшением

- Небольшая разница в высоте карточек.
- Неидеальные переносы, если данные остаются читаемыми.
- Избыточная плотность диагностических блоков owner/admin, если рабочие роли это не видят.
- Текстовые формулировки, которые можно улучшить без риска для логики.

## 7. Результат проверки

После ручного просмотра нужно зафиксировать один из статусов:

- `design_ok` — дизайн-блокеров нет.
- `design_ok_with_notes` — блокеров нет, но есть некритичные улучшения.
- `design_blocked` — есть проблемы, которые нужно исправить до merge.
