# Решения по проекту «Навигатор сделок»

## ADR-001. Репозиторий проекта

Принято: проект «Навигатор сделок» закреплён за репозиторием `deputat36/vktg`.

Причины:

- в репозитории есть страницы `nav-v2.html`, `dashboard-v2.html`, `deals-v2.html`, `deal-card-v2.html`, `spn-v2.html`;
- в `README.md` описан патч ролевого меню для CRM «Навигатор сделок v2»;
- в `config/supabase.js` указано, что Supabase подключён для Навигатора сделки СПН и используется отдельный набор таблиц с префиксом `nav_`.

## ADR-002. Граница с CRM «Лидер»

Принято: «Навигатор сделок» не хранит прикладные данные в таблицах CRM «Лидер».

Запрещено использовать для данных Навигатора:

- `leader_leads`;
- `leader_clients`;
- `leader_orders`;
- `leader_order_items`;
- `leader_catalog`;
- `leader_user_profiles`.

Разрешено использовать общий Supabase Auth, но профили, роли, сделки, участники, документы, расходы и проверки Навигатора должны оставаться в зоне `nav_*`.

## ADR-003. Отображение статусов документов

Принято: в базе и RPC сохраняются технические статусы документов на английском:

- `needed`;
- `received`;
- `checked`.

В интерфейсе пользователю должны показываться русские подписи:

- `Нужен`;
- `Получен`;
- `Проверен`.

Причина: технические значения стабильны для базы и логики, русские подписи нужны для понятного интерфейса обычного сотрудника.

## ADR-004. Вызов функций Навигатора

Принято: функции `nav_*` предназначены для работающего приложения после входа пользователя и для служебных процессов Supabase.

Вызов этих функций до входа пользователя не требуется. Миграция должна сохранять работу существующих сотрудников, не менять тела функций и не затрагивать данные.

Для новых функций `nav_*` необходимо явно назначать требуемые права и не полагаться на стандартные права PostgreSQL.

## ADR-005. Изменение содержимого сделки

Принято: добавление документов, расходов, рисков и задач к сделке разрешено только пользователю, который имеет право редактировать эту сделку.

Причина: просмотр сделки не равен праву менять её содержимое. Юрист, брокер или другой участник может видеть сделку для своей работы, но не должен иметь возможность произвольно добавлять новые сущности без права редактирования сделки.

## ADR-006. Изменение статуса задач

Принято: статус задачи может менять редактор сделки, назначенный исполнитель или сотрудник назначенной роли.

Причина: задача может быть передана юристу, брокеру или другому участнику. Исполнитель должен иметь возможность отметить свою задачу, но не должен менять чужие задачи только потому, что видит сделку.

## ADR-007. Изменение статуса документов

Принято: статус документа может менять редактор сделки, юрист или ипотечный брокер, если они имеют доступ к этой сделке.

Причина: документы — рабочая зона проверки сделки. СПН и менеджер сохраняют полное редактирование через доступ к сделке, а профильные специалисты могут отмечать документы в рамках своей доступной очереди.

## ADR-008. Прямые записи в таблицы задач и событий

Принято: прямые записи в задачи и журнал событий через клиентский доступ закрываются. Изменения должны проходить через защищённые RPC.

Причина: RPC фиксируют автора, проверяют права и создают корректную историю. Прямая запись в таблицу задач или событий может обходить бизнес-логику и подделывать историю действий.

## ADR-009. Изолированный authenticated browser E2E

Принято: authenticated Playwright-тесты Навигатора выполняются только против отдельного Supabase development branch и технических аккаунтов с префиксом `nav-e2e`.

Production project ref блокируется preflight-проверкой. Конфигурация Supabase подменяется только во временном workspace GitHub Actions. В репозиторий и browser bundle не передаются service-role key, JWT, refresh token или приватные access links.

Причина: owner/admin и другие роли не должны получать доступ к реальным клиентским данным ради автоматического теста. Удаление development branch является основным cleanup-механизмом.

## ADR-010. Операционная готовность не равна заполненности анкеты

Принято: операционная готовность вычисляется на сервере read-only и не перезаписывает сохранённые проценты анкеты.

Расчёт обязан учитывать критичные данные сторон, менеджера или документированное исключение, ответственного СПН, распределение профильных специалистов, следующий шаг с владельцем и сроком, блокирующие риски, просроченные обязательные документы и давность активности. Любой критичный пробел ограничивает результат ниже 80%.

Первый этап опубликован как preview для owner/admin/manager; роль viewer подключается позже только к read-only результату и только в пределах уже разрешённых сделок. Изменение реальных назначений, сроков и данных сделки требует отдельного подтверждённого mutation workflow с audit event.

Причина: высокий процент заполнения анкеты не доказывает, что сделка управляемая и может безопасно перейти к задатку или сделке.

## ADR-011. Наблюдатель использует операционную, а не анкетную готовность

Принято: отдельный viewer workspace повторно использует read-only RPC операционной готовности. Роль `viewer` проходит внутренний gate только для сделок, которые уже разрешены `nav_v2_private.nav_v2_can_view_deal(...)`.

Интерфейс наблюдателя показывает статус, готовность, препятствие, ответственных, ближайший срок и ссылку на историю. Назначения, смена статусов, закрытие задач и другие mutation controls отсутствуют.

Причина: отдельный компактный режим не должен возвращать пользователя к вводящему в заблуждение проценту заполненности анкеты или расширять доступ к чужим сделкам.

## ADR-012. Цикл доработки СПН использует единый explicit lifecycle карточки

Принято: возврат СПН, исправление замечаний, повторная отправка и подтверждение результата отображаются одним lifecycle-модулем карточки. Модель состояния остаётся pure, DOM hook получает уже загруженные `deal`, `documents`, `risks`, `tasks`, `comments` и `events`, а mutation выполняется только существующими `nav_v2_return_spn_rework` и `nav_v2_submit_spn_rework`.

Возврат сохраняет роль автора и структурированные категории внутри командного комментария. Для старых карточек без rework events интерфейс явно показывает, что автор возврата не зафиксирован, и строит fallback по текущим пробелам без догадки о человеке. После повторной отправки подтверждение строится по серверному событию после reload, а не по оптимистическому локальному состоянию.

Причина: самостоятельные patch-модули делали повторные запросы карточки, показывали разные списки замечаний и не давали СПН одного проверяемого пути до принятого результата.

## ADR-013. Документная работа юриста строится как один проверяемый цикл

Принято: карточка юриста показывает один приоритетный документ и детерминированное следующее действие в цепочке `нужен → запрошен → получен → проверен / проблема`. Pure-модель использует уже загруженные `documents`, `participants` и `events`; DOM hook не выполняет отдельный read RPC. Изменение проходит только через существующий `nav_v2_update_document_workflow`, а подтверждение после reload строится по серверному событию.

Полный список документов остаётся доступен через progressive disclosure. Проблемные документы, полученные для проверки, просроченные запросы и обязательные документы к задатку/сделке получают объяснимый приоритет. Для каждой строки обязательны сторона, причина, влияние, ответственный, срок, последнее изменение, заметка и следующее действие.

Причина: разрозненные панели запроса, ожидания, проблем и итогового решения заставляли юриста искать контекст по карточке и не показывали единый путь до подтверждённого результата.

## ADR-014. Выполнение подтверждается текущим состоянием и серверным событием

Принято: общий блок результата карточки использует уже загруженную историю и показывает завершение только при совпадении audit event с текущим серверным состоянием задачи, документа, риска или этапа сделки. Повторно открытое, отменённое, no-op или старше семи дней действие не считается актуальным подтверждением.

После подтверждения pure-модель повторно использует action-first расчёт и сразу показывает следующий шаг, ответственного, срок, критерий готовности и прямой переход в рабочий раздел. Имя автора показывается только когда его можно достоверно определить по текущему профилю, назначению сделки или сущности; иначе интерфейс честно сообщает, что имя не передано карточке.

Причина: оптимистическое сообщение после клика не доказывает сохранение результата, а отдельный поиск следующего шага разрывает операционный цикл.

## ADR-015. Мобильный первый экран показывает действие до агрегатов

Принято: на ширине 360–430 px рабочий стол, список сделок, карточка и менеджерский кабинет сначала показывают один операционный результат или ближайшее действие. На первом экране допускается не более трёх кнопок: одна основная и до двух контекстных.

Вторичные приоритеты, карточки, режимы фильтрации, метаданные и агрегатные показатели не удаляются и не подменяются: они переходят в явное progressive disclosure или ниже основной рабочей области. На desktop те же данные остаются развёрнутыми. Перестановка не расширяет ролевые права и не создаёт новых mutation-маршрутов.

Причина: мобильный пользователь должен понять, что сделать дальше, не прокручивая KPI, профиль, фильтры и длинные списки до первого рабочего действия.

## ADR-016. Создание карточки и передача юристу имеют разные gates

Принято: новая анкета СПН имеет ровно три верхнеуровневых этапа. Быстрый черновик требует только режима, стадии, модели сопровождения и объекта либо причины его отсутствия. Формирование рабочей карточки дополнительно требует ведущего СПН, следующего шага и даты либо явной отметки, что дата неизвестна. Handoff юристу является отдельным действием и требует конкретного типа запроса, ожидаемого решения, срока/срочности и структурированного юридического паспорта.

Процент готовности остаётся только вспомогательным индикатором и не заменяет gates. Срочный неполный handoff допускается с явным списком пробелов, но без выдуманных ответов.

Причина: один процент и одна финальная кнопка не позволяют отличить полезный черновик от достаточной передачи юристу.

## ADR-017. Intake rules v1 — versioned pure contract до production-интеграции

Принято: вопросы, триггеры, риски, документы, владельцы и ожидаемые решения задаются versioned-каталогом. Pure JS-модель строит маршрут, юридический паспорт и gates без RPC, `fetch`, Supabase client или browser storage. Значимый факт хранит отдельные `value` и `source`; отсутствие ответа означает `unknown`, а не `no`. Правило по скрытому вопросу не срабатывает.

Автоматическое подключение брокера возможно только для обычной и военной ипотеки. Маткапитал, сертификаты и детские деньги без ипотеки относятся к СПН/юристу. Контракт хранится repository-only и не импортируется production страницами до отдельной проверки presentation layer, SQL prototype, PG17 harness и deployment approval.

Причина: frontend, backend, документы, задачи, паспорт и handoff должны использовать одну проверяемую семантику без расширения production surface на этапе проектирования.

## ADR-018. Юрист получает passport-first экран и честный legacy fallback

Принято: для роли юриста структурированный юридический паспорт v1 показывается перед прежним общим профилем сделки. Первый экран отвечает на вопросы `что требуется`, `к какому сроку`, `что подтверждено`, `что известно со слов клиента`, `что неизвестно`, `какие риски и документы есть`, `что делает СПН`. Кнопки решения переиспользуют существующий юридический lifecycle.

Старая сделка без паспорта не преобразуется молча в подтверждённые факты. Read-only fallback использует только уже загруженные deal/documents/risks/participants, помечает источник как legacy и явно сообщает, что evidence старых фактов не разделён. Массовый backfill не выполняется.

Причина: юрист должен понять обычную ситуацию за 20–30 секунд, но совместимость не должна создавать ложную уверенность или новые персональные данные.

## ADR-019. Документы фильтруются по стороне, а задачи проходят owner gate

Принято: repository-only work plan строится из matched rules единого intake-каталога. Документ продавца или покупателя допускается только когда эта сторона явно сопровождается; документы объекта и сделки допускаются по explicit rule. Для партнёрской сделки сторона не выводится из самого признака партнёрства. Неуместный документ фиксируется как `side_not_accompanied`, но не попадает в создаваемый план.

Каждый task candidate обязан иметь конкретные action, deadline rule, evidence, expected result и gate impact. Роль ответственного сохраняется всегда, но в `ready_tasks` попадают только кандидаты с назначенным owner id. Ипотечный брокер получает только правила `mortgage` и `military_mortgage`; маткапитал и детские деньги остаются отдельной юридической частью.

Причина: автоматически созданная работа не должна относиться к несопровождаемой стороне, а общая задача без ответственного, срока и проверяемого результата не является управляемой задачей.

## ADR-020. Server save не доверяет клиентскому паспорту и плану работ

Принято: будущая server-side интеграция новой анкеты обязана вычислять matched rules, маршрут специалистов, юридический паспорт, side-aware документы, задачи и gates из канонического versioned-каталога. Присланные клиентом `legal_passport`, `intake_work_plan`, owner id и готовность задачи не являются источником истины.

Первый этап остаётся repository-only pure SQL adapter: без DML, migration и публичного RPC. Функции размещаются в private schema, исполняются только server role и проходят detached PostgreSQL 17 harness. Подключение к legacy save, назначение реальных участников и production deployment требуют отдельных branch rehearsal, rollback и явного approval.

Причина: browser preview можно изменить до отправки. Сервер должен повторить бизнес-правила, не расширять брокерский scope за пределы ипотеки, не создавать задачи без проверенного исполнителя и не сохранять запрещённые персональные или документные поля.

## ADR-021. Legacy save нельзя вызывать до закрытия request, owner и parity gates

Принято: новая анкета проходит отдельную pure-цепочку `recompute → exact allowlist → production sanitizer snapshot → legacy call preview`. `client_request_id` передаётся отдельным UUID, а owner assignments — только отдельным trusted server context. Client owner IDs, готовые задачи и паспорт не являются источником истины.

Даже при успешной подготовке `production_call.allowed` остаётся false. Текущий production wrapper не имеет persistent replay ledger; legacy save создаёт generic документы обеих сторон и назначает `auth.uid()` вместо проверенного lead SPN. Из 25 canonical rules 12 не имеют точной legacy semantics. Harness mock разрешён только для доказательства одного business write, exact replay и rollback.

Причина: зелёный pure adapter ещё не делает старую mutation boundary идемпотентной, side-aware и actor-aware. Молчаливое использование legacy создало бы дубли, неверные назначения и документы несопровождаемой стороны.
