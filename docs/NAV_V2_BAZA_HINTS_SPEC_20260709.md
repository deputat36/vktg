# Navigator v2 + BAZA hints spec — 2026-07-09

## Цель

Описать безопасный первый формат подсказок BAZA для карточки сделки Navigator v2. Документ дополняет `docs/NAV_V2_BAZA_READONLY_BOUNDARIES_20260705.md` и не меняет live-логику.

## Базовый принцип

BAZA даёт только справочные подсказки. Navigator v2 остаётся источником рабочих данных по сделке. Подсказки не должны записывать данные в BAZA и не должны отправлять туда персональные сведения клиента или сотрудников.

## Формат подсказки

Минимальный формат approved hint:

```json
{
  "id": "mortgage-docs-basic",
  "title": "Проверьте пакет документов по ипотеке",
  "role": "spn",
  "reason": "В сделке отмечена ипотека",
  "body": "Кратко напомните клиенту список документов и сроки передачи юристу/брокеру.",
  "material_url": "",
  "priority": "normal",
  "is_active": true
}
```

## Разрешённые поля

- `id` — стабильный технический ID подсказки.
- `title` — короткий заголовок.
- `role` — для кого подсказка: `spn`, `lawyer`, `broker`, `manager`, `owner`, `admin`, `all`.
- `reason` — почему подсказка показана.
- `body` — короткое содержание без персональных данных.
- `material_url` — ссылка на утверждённый материал, если есть.
- `priority` — `low`, `normal`, `high`.
- `is_active` — включена ли подсказка.

## Разрешённые сигналы сделки

Подсказки можно выбирать только по обезличенным сигналам:

- тип объекта;
- стадия сделки;
- наличие ипотеки;
- наличие материнского капитала;
- наличие сертификата;
- тип юридического риска без ФИО и адреса;
- статус документов;
- роль текущего пользователя.

## Запрещённые данные

В BAZA и в hint matching нельзя передавать:

- ФИО клиента;
- телефон клиента;
- полный адрес объекта;
- паспортные данные;
- банковские данные;
- договорные номера;
- персональные комментарии сотрудников;
- внутренние оценки конкретных людей.

## Первый UI-блок

Название блока в карточке сделки:

`Подсказки по сделке`

В первой версии показывать максимум 3–5 подсказок:

- заголовок;
- короткая причина показа;
- роль;
- ссылка на материал, если есть.

Подсказки должны быть read-only. Никаких кнопок `применить`, `изменить сделку`, `отправить в BAZA` на первом этапе.

## Минимальный алгоритм подбора

1. Собрать обезличенные сигналы сделки на frontend или в отдельном read-only endpoint.
2. Не включать ФИО, телефон, адрес и комментарии.
3. Сопоставить сигналы с approved hint list.
4. Отфильтровать подсказки по роли пользователя.
5. Отсортировать по priority.
6. Показать не более 5 подсказок.

## MVP без базы данных

Для первого безопасного прототипа можно использовать статический JSON-файл в репозитории:

`assets/data/nav-v2/baza-hints.json`

Плюсы:

- нет записи в Supabase;
- нет доступа к персональным данным;
- проще проверить UI;
- легко откатить.

Минусы:

- обновление подсказок требует commit/deploy;
- нет удобной админки.

## Следующий этап после MVP

После ручной проверки можно выбрать один из вариантов:

1. приватные `kb_*` таблицы в Supabase только для approved knowledge;
2. отдельный read-only endpoint BAZA;
3. оставить статический JSON, если подсказок мало.

## Критерии готовности MVP

- Подсказки отображаются в карточке сделки.
- Подсказки не содержат персональные данные.
- Блок скрывается, если подходящих подсказок нет.
- Обычные роли видят только свои подсказки.
- Owner/admin могут видеть все подсказки для проверки.
- Нет записи из Navigator v2 в BAZA.
- Нет изменений в `leader_*`, `parket_*` и legacy Navigator.
