# Navigator v2 — краткая запись для CRM

Дата: 23 июля 2026 года  
Статус: frontend read-only  
Production Supabase: без изменений

## Зачем нужен блок

Navigator не должен становиться второй CRM. Его задача — проанализировать состояние подготовки сделки и помочь сотруднику зафиксировать в основной CRM только содержательный итог.

В карточку сделки добавлен блок `В CRM`, который формирует готовую процессную запись:

- текущий этап;
- подтверждённый результат по состоянию карточки;
- риск или препятствие;
- договорённость по расходам и расчётам;
- недостающие обязательные пункты;
- следующее действие с ответственным и сроком.

## Что блок не делает

- не записывает данные в CRM автоматически;
- не меняет Navigator или Supabase;
- не вызывает RPC, Edge Function или REST-запрос;
- не использует localStorage или sessionStorage;
- не создаёт отдельный журнал;
- не копирует адрес объекта, ФИО сторон, телефон, email, паспортные или иные идентифицирующие сведения;
- не использует свободное поле `next_action`, потому что оно может содержать лишние персональные детали;
- не заменяет решение юриста, брокера или менеджера.

## Принцип формирования

Источник — только уже загруженное процессное состояние карточки:

`статус → открытые документы → открытые задачи → блокирующие риски/решения → согласование расходов/расчётов → ближайшее действие`

Приоритет следующего действия:

1. блокирующее решение профильного специалиста;
2. блокирующий риск;
3. обязательный документ;
4. ближайшая открытая задача;
5. согласование расчётов или расходов;
6. пострегистрационные действия;
7. безопасный общий fallback.

Если ответственный или срок не определены, блок сообщает это явно. Он не придумывает дату и не скрывает пробел.

## Пример записи

```text
Текущий этап: подготовка недостающих документов
Результат: Готовность: к задатку 55%, к сделке 35%. Открыто: 1 документов, 0 задач, 0 блокирующих пунктов.
Риск или препятствие: Требуют решения или устранения: обязательные документы: 1.
Договорённость: Расходы согласованы; порядок расчётов ещё требует согласования.
Не хватает: 1 обязательных документов.
Следующее действие: СПН: получить обязательный документ и передать его профильному специалисту на проверку; 25.07.2026.
```

Сотрудник обязан проверить запись перед сохранением в CRM, особенно ответственного, срок и актуальность договорённостей.

## Проверки

Контракт и CI подтверждают:

- fixed compact structure;
- отсутствие клиентских идентификаторов даже при их наличии во входных данных;
- исключение подтверждённых `not_applicable/replaced/cancelled` документов из активного долга;
- исключение подтверждённых terminal outcome рисков из блокеров;
- явного ответственного и срока либо отметки, что срок требуется уточнить;
- отсутствие network, storage и mutation surface;
- интеграцию через существующий consolidated deal-card enhancement hook без нового standalone HTML-модуля.

## Boundary

Изменение относится только к frontend source, fixtures, unit tests, checker, workflow и документации.

Не выполнялись:

- Supabase migration apply/repair;
- SQL mutation;
- Auth/RLS/grants changes;
- Edge deploy;
- preview branch creation;
- technical account or secret creation;
- production data cleanup;
- изменения `leader_*`.

Решение:

`crm_handoff_summary_added_read_only_no_crm_write`
