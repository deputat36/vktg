# Navigator v2 — repository-only DTO prototype списка сделок

Дата: 16 июля 2026 года  
RPC: `public.nav_v2_get_deals_list(p_limit integer)`  
Production deploy: не выполнялся

## Цель

Текущий production RPC возвращает клиентские ФИО и телефоны, исходный legacy title, точный адрес и свободный `next_action`. Frontend уже маскирует часть этих данных, но серверный контракт остаётся избыточным.

Прототип в `supabase/prototypes/nav_v2_get_deals_list_explicit_dto.sql` задаёт явный allowlist и сохраняет рабочую ценность списка:

- статус и уровень риска;
- готовность к задатку и сделке;
- задачи, документы, риски и сроки;
- последнюю активность;
- назначенных сотрудников;
- безопасный ориентир объекта;
- контролируемую системную подсказку следующего действия.

Это repository-only работа без production deploy, изменения grants, RLS, Auth или Edge Functions.

## Что исключается

DTO не возвращает:

- `seller_name`, `buyer_name`;
- `seller_phone`, `buyer_phone`;
- кадастровый номер;
- `deal_summary`;
- `wizard_snapshot`;
- комментарии и review-тексты;
- исходный пользовательский `next_action`;
- номер квартиры, комнаты, офиса или помещения;
- полную строку `nav_deals_v2`.

## Безопасные производные поля

### Нейтральная ссылка

`title` и `display_title` сохраняются для совместимости, но содержат только:

`тип объекта — безопасный ориентир · короткий ID`

Для демо-карточки добавляется префикс `ДЕМО:`. Исходный title не возвращается.

### Адрес

`address` содержит ориентир без unit-level данных. Номер квартиры, комнаты, офиса или помещения удаляется серверной функцией.

### Следующее действие

Ключ `next_action` сохраняется для существующих интерфейсов, но его значение вычисляется из контролируемых фактов:

- возврат на доработку;
- красные риски;
- просроченные задачи;
- недостающие документы;
- отсутствие юриста или ипотечного брокера;
- несогласованные расчёты или расходы;
- готовность к задатку;
- факт наличия записанного шага.

Исходный свободный текст не выдаётся. Дополнительно возвращается `has_recorded_next_action` — только булев факт.

### Восстановление сохранения

`created_by_current_user` и `preparation_mode` позволяют СПН найти только что созданную карточку после неопределённого сетевого ответа без сравнения точного адреса клиента.

Текущий frontend подготовлен к обоим контрактам:

- со старым production DTO используется прежний точный address fallback;
- с новым DTO используются автор, тип объекта, режим подготовки и время создания.

## Роль ипотечного брокера

Прототип учитывает исправленную модель ответственности после PR #349:

- `broker_needed` относится только к ипотеке и военной ипотеке;
- брокер консультирует, подбирает программу, помогает получить одобрение и обучает СПН;
- маткапитал и сертификаты остаются в контуре СПН и юриста;
- оформление ипотечной сделки выполняют СПН и юрист.

Контролируемая подсказка DTO поэтому предлагает назначить ипотечного брокера только когда `broker_needed = true`.

## Ролевая семантика

Прототип сохраняет существующее правило видимости:

- owner/admin видят все доступные карточки;
- создатель, СПН сторон, менеджер, назначенные юрист и брокер видят свою сделку;
- юрист видит сделки с `lawyer_needed = true`;
- ипотечный брокер видит сделки с `broker_needed = true`;
- участник видит карточку через `nav_deal_participants_v2`.

Public signature и EXECUTE grants не меняются.

## Проверки

CI должен подтвердить:

- SQL находится только в `supabase/prototypes`;
- `production_applied = false`;
- JSON allowlist совпадает с SQL;
- отсутствует `select d.*` и full-row serialization;
- client identifier keys не выдаются;
- title, address и next action формируются безопасно;
- consumer matrix соответствует фактическим frontend-полям;
- save recovery поддерживает старый и новый контракт;
- handoff fallback не зависит от клиентских ФИО и телефонов;
- изменённые JS-файлы проходят syntax checks.

## Production gate

Применять prototype как migration можно только после:

1. authenticated-smoke для owner/admin/manager/SPN/lawyer/broker;
2. проверки desktop и mobile;
3. сравнения response shape старого и нового RPC на синтетических данных;
4. проверки восстановления сохранения;
5. проверки фильтров, dashboard priority, lawyer и broker work modes;
6. готового rollback SQL.

Платную preview branch без отдельного решения владельца не создавать.

## Rollback

Rollback должен восстановить production function body, полученный через `pg_get_functiondef('public.nav_v2_get_deals_list(integer)'::regprocedure)`, без изменения сигнатуры и grants.

Каноническая migration chain текущей функции заканчивается файлом:

`supabase/migrations/20260625095500_navigator_deals_list_add_lawyer_broker_names.sql`

Frontend подготовлен к старому контракту, поэтому откат сервера не требует срочного отката интерфейса.

## Результат волны

- явный server DTO спроектирован;
- client PII и raw free text исключены из выходного контракта;
- потребители проинвентаризированы;
- критические потребители подготовлены;
- production Supabase не изменён;
- следующий этап после merge — terminal/exception states документов и рисков либо authenticated DTO validation после одобрения среды.
