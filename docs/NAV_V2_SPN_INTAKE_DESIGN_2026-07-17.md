# Navigator v2 — проект трёхэтапного создания сделки

Дата: 17 июля 2026 года. Статус: repository-only contract v1; production mutations не подключены.

## 1. Целевой путь

| Этап | Вопрос пользователя | Содержимое | Основное действие |
|---|---|---|---|
| 1. Что происходит | «Как быстро зафиксировать ситуацию?» | Режим, сопровождение, стадия, объект/причина отсутствия, адрес/ориентир, дата/срочность, ведущие СПН | `Продолжить` |
| 2. Что важно проверить | «Какие факты реально меняют сценарий?» | Только активные группы: продавец, покупатель, объект, деньги/условия, специальные обстоятельства | `Проверить итог` |
| 3. Проверить и сохранить | «Что сохранится и кому это нужно передать?» | Краткая карточка, известное/неизвестное, риски, документы, следующий шаг, специалисты, запрос и ожидаемое решение | Одна выбранная primary action |

Черновик и handoff — разные состояния. Создание карточки никогда само по себе не означает передачу юристу.

## 2. Wireframe-контракт

### Экран 1

| Зона | Правило |
|---|---|
| Верхняя строка | `Этап 1 из 3`, название и статус автосохранения локального черновика |
| Выбор режима | 5 вариантов; один visual block |
| Сопровождение | 6 вариантов; при двух СПН раскрывается назначение по сторонам |
| Стадия | Не более 7 вариантов; отдельная срочность/дата |
| Объект | Один список типов; `объект ещё не выбран` требует понятную причину |
| Footer | Одна primary-кнопка `Продолжить`; `Сохранить черновик` — спокойная secondary action |

### Экран 2

Каждый активный факт — один короткий вопрос с ответами `Да / Нет / Не знаю / Не относится`. Если ответ юридически значим, ниже появляется источник `Документ / Со слов клиента / Пока не проверено`. Неактивный вопрос не рендерится и не запускает правило.

Заполненные группы сворачиваются в строку `название · отвечено N из M`; изменение раннего ответа пересчитывает группы без очистки скрытых данных. Перед удалением несовместимых данных требуется отдельное подтверждение.

### Экран 3

| Карточка | Что показывает |
|---|---|
| Создаётся | Режим, стадия, объект, сопровождение, СПН, дата |
| Известно | Отдельно `подтверждено документом` и `со слов клиента` |
| Неизвестно | Явный список, а не молчаливое «нет» |
| Проверки | Риски, блокировка задатка/сделки, требуемые статусы документов |
| Следующий шаг | Готовый вариант + дополнительная заметка |
| Маршрут | Юрист; брокер только для обычной/военной ипотеки |
| Обращение к юристу | Тип запроса, ожидаемое решение, срок/срочность |

Primary action выбирается контекстом: обычный flow — `Сохранить и продолжить самостоятельно`; при подтверждённом handoff — `Сохранить и передать юристу`; при чистом ипотечном маршруте — `Сохранить и передать брокеру`. Остальные действия доступны через компактное раскрытие, а не как четыре равнозначные кнопки.

## 3. Source contract

Канонические repository-only файлы первой версии:

- `config/nav-v2-intake-contract-v1.json` — этапы, вопросы, справочники и юридические правила;
- `assets/js/nav-v2/spn-intake-contract-v1.js` — pure route/assessment model;
- `tests/fixtures/nav-v2-intake-contract-v1.json` — сценарии;
- `scripts/check-nav-v2-intake-contract-v1.mjs` — semantic contract;
- `scripts/check_nav_v2_intake_contract_v1.py` — source/privacy/deployment guard.

Контракт не содержит `fetch`, RPC, Supabase client и browser storage. Это позволяет проверять UI и смысловые правила без новых mutation-запросов.

## 4. Модель факта

```json
{
  "minor_seller": {
    "value": "yes",
    "source": "client"
  }
}
```

Допустимые значения: `yes`, `no`, `unknown`, `not_applicable`. Допустимые источники: `document`, `client`, `unchecked`. Неизвестное, невалидное или не подтверждённое значение не превращается в `no`.

## 6. Gates вместо единого процента

| Gate | Минимум | Результат |
|---|---|---|
| `save_draft` | режим, стадия, сопровождение, объект или причина отсутствия | Можно сохранить черновик; handoff не создаётся |
| `form_card` | draft gate + ведущий СПН + следующий шаг + дата или `дата не определена` | Можно сформировать рабочую карточку |
| `handoff_lawyer` | тип запроса, ожидаемое решение, срок/срочность, объект, сопровождение, известные/неизвестные факты, риски, статусы документов, следующий шаг | `ready`, `urgent_incomplete` или `blocked` |

`urgent_incomplete` допустим только для срочного кейса без критических пробелов. UI обязан перечислить недостающее; пользователь не должен выдумывать ответы.

## 7. Единое правило

Каждая запись каталога описывает цепочку:

`trigger → risk level → deposit/deal gate → required documents → owner → request type → expected decision`.

Правила с trigger по факту работают только если соответствующий вопрос активен для текущего сценария. Так скрытый вопрос не может создать риск или маршрут.

Broker scope жёстко ограничен `mortgage` и `military_mortgage`. Маткапитал, сертификат, субсидия и детские деньги без ипотеки маршрутизируются к СПН/юристу. При сочетании ипотеки и маткапитала брокер получает только ипотечную часть.

## 8. Side-aware документы

Repository-only реализация: `assets/js/nav-v2/spn-intake-work-plan-v1.js`. Модель получает уже matched rules, возвращает `document_candidates` и отдельно `skipped_documents`; это не mutation DTO и не создаёт строк в production.

Будущий генератор формирует не файлы, а безопасные рабочие записи: `type`, `side`, `status`, `owner`, `deadline`, `allowed_link`, `reason`, `gate_impact`.

| Scope | Правило |
|---|---|
| Документ сопровождаемой стороны | Создавать только если `representation_model` включает эту сторону |
| Общий объектный документ | Допустим для обеих моделей, если объективно нужен для сделки; `side=object` |
| Документ второй стороны | Не создавать только «на всякий случай»; требуется explicit rule с причиной |
| Партнёрская сделка | Указывать owner `partner/spn` и границу ответственности |
| Trigger document | Создавать только из matched rule; фиксировать, какой риск и gate его потребовали |

Navigator хранит статус и разрешённую ссылку. Загрузка файла или скана в Navigator не является частью контракта.

## 9. Конкретные задачи

Repository-only реализация возвращает `task_candidates` и `ready_tasks`. Кандидат без owner id остаётся `needs_owner`; будущая интеграция не должна создавать его до назначения ответственного.

Автоматическая задача допустима только при полном completion contract:

```text
owner + action + deadline rule + evidence + expected result + gate impact
```

Примеры:

| Trigger | Owner | Action | Evidence | Expected result |
|---|---|---|---|---|
| Доверенность | lawyer | Проверить полномочия на подписание и получение денег | Статус проверки доверенности | `можно / нужна новая / стоп` |
| Несовершеннолетний собственник | lawyer | Определить действия по опеке до задатка | Решение юриста + статус разрешения | Допустимый порядок действий |
| Расчёты не согласованы | spn | Согласовать способ, срок и условие раскрытия | Структурированный статус расчётов | `agreed` или открытый вопрос |
| Обычная ипотека | broker | Проверить программу и статус одобрения | Статус одобрения | Ипотечная часть готова/не готова |

Общая задача «проверить документы» без владельца, срока и результата не создаётся.

## 10. Первый экран юриста

Порядок чтения рассчитан на 20–30 секунд:

1. Заголовок: конкретный request type и ожидаемое решение.
2. Срочность/дата, режим и стадия.
3. Объект, сопровождение и СПН по сторонам.
4. Подтверждённые факты.
5. Факты со слов клиента.
6. Неизвестные факты.
7. Стоп-факторы и потенциальные риски.
8. Четыре статуса документов.
9. Расчёты, расходы, задаток и следующий шаг СПН.
10. Решение: `можно дальше / нужна информация / нужны документы / стоп-фактор / вернуть СПН`.

Сырой `wizard_snapshot` остаётся диагностическим источником, но не требуется для обычного решения.

## 11. Backward compatibility

- Сделка с `legal_passport.version=1` показывается напрямую.
- Старый snapshot проходит через `adaptLegacyWizardDraft()` и получает preview паспорта без записи в базу.
- Legacy flags без явного отрицательного ответа преобразуются в `unknown`, а не `no`.
- Старые маршруты сохранения, duplicate guard и idempotency request ID не отключаются до отдельной интеграции и PG17 regression.
- Массовый backfill старых сделок запрещён; паспорт создаётся при новом handoff или явном редактировании.
- Неизвестная будущая версия паспорта показывает безопасный fallback, а не интерпретируется как v1.

## 12. План интеграции

1. Repository-only contract и semantic fixtures.
2. Detached presentation layer: три этапа, local draft recovery, preview паспорта; zero mutation.
3. Detached legal passport block в карточке/очереди с legacy adapter.
4. SQL prototype и PG17 harness: единый каталог, side-aware documents, конкретные tasks, rollback; production apply запрещён.
5. После отдельного deployment approval — совместимое подключение к save RPC, pilot и наблюдение без PII.

## 14. Rollback

На текущем этапе rollback — удалить новые repository-only файлы и workflow: они не импортированы production страницами и не меняют runtime. На будущей интеграции rollout должен быть feature-flagged; отключение флага возвращает существующий мастер и карточку без изменения сохранённых legacy данных.
