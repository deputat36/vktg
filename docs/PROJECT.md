## Границы проекта

«Навигатор сделок» может использовать общий Supabase Auth с CRM «Лидер», но прикладные данные проекта должны храниться отдельно.

Разрешённая зона данных проекта:

- таблицы, функции, политики и представления с префиксом `nav_`;
- собственные профили, роли и участники Навигатора: `nav_user_profiles`, `nav_roles`, `nav_members` или их актуальные аналоги.

Запрещено использовать для прикладных данных Навигатора:

- `leader_leads`;
- `leader_clients`;
- `leader_orders`;
- `leader_order_items`;
- `leader_catalog`;
- `leader_user_profiles`.

Данные проектов нельзя смешивать.

## Архитектурные правила

- Все изменения базы выполнять через миграции.

- Перед задачей проверять GitHub, Supabase, основную ветку, свежие PR и текущую реализацию.
- Минимально менять существующий код.
- Не ломать работающие роли: СПН, юрист, ипотечный брокер, менеджер, администратор/owner.
- Проверять разграничение прав и RLS при изменениях данных.
- Не создавать дублирующие таблицы и сущности без необходимости.

## Текущий рабочий контекст

13–14 июня 2026 устранена проблема прыжка страницы вверх в карточке сделки при изменении статуса документа. Для этого был создан и смержен PR #10, где сохранение статуса документа обрабатывается без полной перезагрузки карточки.

Текущая небольшая задача: после перезагрузки карточки показывать статусы документов по-русски, не меняя технические значения в базе:

- `needed` → `Нужен`
- `received` → `Получен`
- `checked` → `Проверен`
