# Navigator v2 — lightweight consultation lifecycle prototype

Дата: 16 июля 2026 года.

Статус: repository-only prototype. SQL не применён к production Supabase.

## Файлы и порядок

Prototype состоит из двух последовательных файлов:

1. `supabase/prototypes/nav_v2_consultation_lifecycle.sql` — базовая схема и lifecycle;
2. `supabase/prototypes/nav_v2_consultation_lifecycle_hardening.sql` — Hardening overlay.

В изолированных тестах их необходимо применять только в указанном порядке. Будущая production migration должна объединить итоговое состояние, а не копировать оба файла вслепую.

## Зачем нужен отдельный контур

Обычный юридический вопрос не должен превращаться в незавершённую сделку с преждевременно созданными документами, рисками и задачами.

Целевой путь:

`вопрос СПН → минимальные факты → список/очередь → ответ или уточнение → при необходимости convert_to_preparation → закрытие`

Полный lifecycle:

`new → need_info / answered / convert_to_preparation → closed`

Незавершённый вопрос также можно перевести в `cancelled`.

## Что проектируется

### Таблицы

- `nav_consultations_v2` — безопасные структурированные метаданные запроса;
- `nav_consultation_messages_v2` — вопрос, запрос уточнения, уточнение, ответ и рекомендация полной подготовки.

Контур не создаёт сделку, задачу, документ или риск.

### Публичные RPC

- `nav_v2_create_consultation`;
- `nav_v2_get_consultation_queue`;
- `nav_v2_get_consultation`;
- `nav_v2_decide_consultation` с обязательным `conversion_mode` для преобразования;
- `nav_v2_add_consultation_clarification`;
- `nav_v2_close_consultation`.

## Hardening overlay

Overlay устраняет пять проблем базового prototype.

### 1. Идемпотентное создание

Create payload обязан содержать `client_request_id` UUID.

Пара `(created_by, client_request_id)` защищена уникальным partial index. Повторный запрос с тем же ключом возвращает существующую консультацию с `idempotent=true` и не добавляет второе сообщение.

### 2. Explicit payload allowlist

Create RPC принимает только:

- `client_request_id`;
- `question`;
- `safe_reference`;
- `request_type`;
- `representation_model`;
- `object_type`;
- `stage`;
- `funding_sources`;
- `circumstances`;
- `planned_event_date`;
- `has_external_documents`.

Любой неизвестный ключ отклоняется. Это исключает молчаливое игнорирование случайно переданных client fields.

### 3. Собственный список СПН

`nav_v2_get_consultation_queue` становится общим role-scoped списком:

- СПН видит только созданные им активные консультации;
- менеджер — консультации своей команды;
- юрист — рабочую очередь;
- owner/admin — весь активный контур;
- broker/viewer — без доступа.

### 4. Историческая юридическая видимость

Неназначенный юрист видит только открытые статусы `new` и `need_info`.

Он не может открыть по прямому UUID неназначенную консультацию в `answered`, `convert_to_preparation`, `closed` или `cancelled`.

Назначенный юрист сохраняет доступ к обработанной им консультации.

### 5. Явный режим преобразования

Для `convert_to_preparation` юрист обязан указать:

- `deposit`; или
- `deal`.

`conversion_mode` запрещён для решений `answer` и `need_info`.

Conversion draft больше не угадывает режим по стадии и не возвращает `unknown`.

## Ролевые правила

### СПН

- создаёт запрос;
- получает список только собственных запросов;
- читает собственную карточку;
- отвечает на `need_info`;
- закрывает ответ или отменяет незавершённый запрос;
- не принимает юридическое решение.

### Менеджер

- может создать запрос;
- видит запросы своих СПН;
- не заменяет юриста и не принимает решение `answer/need_info/convert_to_preparation`.

### Юрист

- видит неназначенные открытые запросы и запросы, назначенные ему;
- отвечает, запрашивает уточнение или рекомендует полную подготовку;
- при первом решении неназначенный запрос закрепляется за этим юристом;
- не получает доступ к исторической неназначенной карточке по UUID.

### Owner/admin

- видит весь контур;
- может принять решение и закрыть запрос с аудитом через историю сообщений.

### Ипотечный брокер

Не получает доступ к юридическому диалогу.

При ипотеке или военной ипотеке create RPC возвращает только признак параллельного брокерского маршрута. Ипотечный брокер отвечает за ипотечную консультацию, программу и одобрение.

Маткапитал и сертификат без ипотеки остаются в контуре СПН и юриста. При сочетании ипотеки с маткапиталом или сертификатом брокер ведёт только ипотечную часть.

### Viewer

Не видит и не изменяет консультации.

## Минимальный DTO списка

Список не возвращает полный вопрос или переписку. Доступны только:

- нейтральная ссылка `Консультация XXXXXXXX`;
- статус и тип запроса;
- сторона сопровождения, тип объекта и стадия;
- источники средств;
- количество особых обстоятельств;
- плановая дата;
- наличие внешних документов без ссылки;
- сотрудник-инициатор и назначенный юрист;
- количество сообщений, возраст, приоритет и ближайшее действие.

Полный текст читается только через detail RPC после ролевой проверки.

## Privacy

Новый контур не принимает отдельные поля:

- ФИО продавца или покупателя;
- телефон и email;
- паспорт, СНИЛС и номер карты;
- кадастровый номер;
- полный адрес и номер квартиры, комнаты, офиса или помещения.

Текстовый guard отклоняет:

- email;
- телефон;
- паспортные последовательности;
- СНИЛС;
- кадастровые номера;
- unit-level номера;
- возможные ФИО;
- длинные платёжные номера.

Это защита от очевидных ошибок, а не доказательство полной деидентификации. В интерфейсе должна оставаться явная инструкция не вводить персональные данные.

## Внешние документы

До решения владельца по утверждённым доменам и retention rules система хранит только `has_external_documents=true/false`.

Сам document URL в prototype не сохраняется. Ссылка продолжает передаваться по утверждённому внешнему каналу.

## Решения юриста

### `answer`

Юрист даёт ответ. Статус становится `answered`. `conversion_mode` должен отсутствовать.

### `need_info`

Юрист формулирует, какого факта не хватает. Статус становится `need_info`. После уточнения СПН запрос возвращается в `new`. `conversion_mode` должен отсутствовать.

### `convert_to_preparation`

Юрист подтверждает необходимость полного процесса и выбирает `deposit` или `deal`. RPC возвращает безопасный `conversion_draft`.

Сделка и backlog автоматически не создаются. Пользователь должен явно открыть полный мастер и подтвердить перенос.

## Security

Эффективное состояние определяется после base + Hardening overlay:

- RLS включён на обеих таблицах;
- прямые права `anon` и `authenticated` на таблицы отозваны;
- `PUBLIC`, `anon` и `authenticated` не могут выполнять prototype RPC;
- EXECUTE оставлен только `service_role` для изолированной проверки;
- private helpers недоступны `authenticated`;
- каждый SECURITY DEFINER RPC проверяет `auth.uid()` и активный профиль;
- функции используют пустой `search_path` и полные имена схем;
- authenticated grants откладываются до отдельной deploy-миграции.

SECURITY DEFINER является API-периметром. Production rollout требует отдельного review минимальных grants и владельцев функций.

## Что не меняется

- production Supabase;
- существующие сделки, комментарии, review, документы, риски и задачи;
- `nav_v2_get_lawyer_queue` для полных сделок;
- роли сотрудников и назначения;
- Auth, Edge Functions и текущие production RLS-политики;
- production readiness и status guards.

## Production gate

До применения обязательны:

1. executable PostgreSQL 17 tests для base + overlay;
2. изолированные authenticated role/mutation tests для СПН, менеджера, юриста, брокера, viewer и owner/admin;
3. проверка идемпотентности и unknown payload rejection;
4. проверка всех переходов и запретов на чужие запросы;
5. negative tests для персональных данных;
6. проверка ACL после обоих SQL-файлов;
7. Security Advisor после DDL;
8. решение владельца по источникам внешних документов и retention rules;
9. rollback rehearsal;
10. отдельный review production migration с минимальными grants.

## Rollback

Так как SQL находится только в `supabase/prototypes`, production rollback не требуется.

В изолированной среде rollback должен:

1. отозвать EXECUTE у consultation RPC;
2. удалить четырёхаргументный и возможный старый трёхаргументный decide RPC;
3. удалить публичные и private functions;
4. удалить unique index идемпотентности;
5. удалить индексы базового prototype;
6. удалить `nav_consultation_messages_v2`;
7. удалить `nav_consultations_v2`;
8. подтвердить отсутствие изменений в `nav_deals_v2`, задачах, документах и рисках.
