# Navigator v2 — серверный prototype консультаций

Дата: 16 июля 2026 года.

## Статус

`repository-only prototype`

SQL находится в `supabase/prototypes/nav_v2_consultation_entity.sql` и не применён к production Supabase.

## Задача

Быстрая консультация должна стать отдельным коротким процессом, а не незавершённой сделкой с десятками автоматически созданных документов и задач.

Целевой lifecycle:

`new → need_info → new → answered → converted / closed`

Юрист принимает одно из трёх решений:

- `answer` — вопрос закрыт ответом;
- `need_info` — нужны уточнения от СПН;
- `convert_to_preparation` — нужно перейти к подготовке задатка или сделки.

## Таблицы

### `nav_consultations_v2`

Хранит:

- автора запроса;
- idempotency key `client_request_id`;
- статус и приоритет;
- безопасный структурированный intake;
- назначенного юриста;
- последнее решение;
- режим преобразования;
- факт запроса преобразования;
- связанную сделку только после явной привязки;
- причину закрытия и timestamps.

Таблица не содержит:

- ФИО продавца или покупателя;
- телефонов и email клиентов;
- паспортов и СНИЛС;
- адреса с номером квартиры;
- кадастрового номера;
- `wizard_snapshot` или `deal_summary`.

### `nav_consultation_messages_v2`

Append-only диалог:

- вопрос юриста на уточнение;
- ответ СПН;
- ответ юриста;
- инструкция по преобразованию;
- причина закрытия.

Редактирование и удаление сообщений RPC-контрактом не предусмотрено.

### `nav_consultation_events_v2`

Append-only аудит без копирования свободного текста. Event data содержит только коды, статусы, режим преобразования и безопасные идентификаторы внутренних сущностей.

## DTO

### Список

Список возвращает только:

- внутренний ID;
- статус и приоритет;
- безопасный preview вопроса;
- сторону, стадию и тип объекта;
- безопасный ориентир;
- коды финансирования и обстоятельств;
- дату;
- признак ипотечного scope;
- состояние назначения юриста;
- имена сотрудников;
- decision/conversion flags;
- timestamps.

В списке отсутствуют полный ответ, известные факты и ссылка на документы.

### Карточка

Полный вопрос, безопасные известные факты, HTTPS-ссылка и ответ доступны только после проверки видимости конкретной консультации.

## Доступ

### СПН

Видит созданные им консультации и отвечает на `need_info`.

### Менеджер

Видит консультации СПН, у которых `manager_id` равен ID менеджера. Может помочь ответить и выполнить явное преобразование.

### Юрист

Видит:

- неназначенную очередь `new/need_info`;
- назначенные ему консультации;
- консультации, в которых он уже оставлял сообщения.

Первое решение автоматически закрепляет консультацию за юристом. Другой юрист не может перехватить назначенную консультацию без owner/admin.

### Owner/admin

Видит и управляет всем consultation-контуром.

### Ипотечный брокер

Не получает доступ к юридической очереди или юридическому ответу.

Поле `broker_scope_needed` означает только необходимость параллельной ипотечной консультации при:

- ипотеке;
- военной ипотеке.

Маткапитал и сертификат без ипотеки не включают брокера. При сочетании с ипотекой ипотечный брокер отвечает только за программу, консультацию и одобрение банка; правовая и расчётная схема и оформление сделки остаются у СПН и юриста.

### Viewer

Доступа нет.

## Server-side privacy

Frontend guard недостаточен, поэтому SQL prototype повторно проверяет:

- email;
- телефон;
- паспортные последовательности;
- СНИЛС;
- номера карт;
- кадастровый номер;
- номер квартиры, комнаты, офиса или помещения;
- возможные ФИО;
- HTTPS-схему ссылки.

Create RPC принимает только explicit allowlist полей и отклоняет неизвестные ключи.

## RPC

- `nav_v2_create_consultation` — idempotent create.
- `nav_v2_get_consultations_list` — role-scoped list DTO.
- `nav_v2_get_consultation_card` — role-scoped card DTO.
- `nav_v2_decide_consultation` — решение юриста.
- `nav_v2_reply_consultation` — уточнение от СПН/менеджера.
- `nav_v2_request_consultation_conversion` — безопасный wizard draft.
- `nav_v2_bind_consultation_conversion` — привязка уже созданной сделки.
- `nav_v2_close_consultation` — закрытие с причиной.

## Преобразование без скрытого backlog

`nav_v2_request_consultation_conversion`:

- требует решение юриста `convert_to_preparation`;
- возвращает безопасный draft для существующего мастера;
- не создаёт сделку;
- не создаёт задачи;
- не создаёт документы;
- не создаёт риски;
- не меняет статус на `converted`.

Пользователь явно проходит существующий мастер. Только после фактического создания доступной ему сделки вызывается bind RPC, и консультация получает статус `converted`.

## Security boundary

- RLS включён на всех трёх таблицах.
- Политики прямого доступа отсутствуют: deny-by-default.
- Прямые grants таблицам не добавляются.
- Все новые функции явно `REVOKE` у `PUBLIC`, `anon` и `authenticated`.
- GRANT statements отсутствуют.
- Production deploy должен отдельно решить, какие reviewed RPC получают `EXECUTE`.

Такая схема нужна потому, что SECURITY DEFINER-функция с EXECUTE для authenticated является полноценным API endpoint и обязана иметь строгую проверку роли и входов.

## Синтетические проверки

Fixtures проверяют:

- доступ создателя и его менеджера;
- запрет другому СПН и чужому менеджеру;
- неназначенную и назначенную очередь юриста;
- запрет перехвата другим юристом;
- отсутствие доступа broker/viewer;
- `need_info → reply → new`;
- обычный ответ;
- решение о преобразовании;
- запрос draft без создания backlog;
- запрет bind до запроса;
- статус `converted` только после привязки сделки;
- запрет решения брокером;
- закрытие и запрет закрывать converted consultation;
- ипотеку, военную ипотеку, маткапитал и сертификаты.

## Production gate

До применения нужны:

1. выполнение SQL в изолированной synthetic-среде;
2. проверка DDL и типов Postgres;
3. authenticated role/mutation E2E для каждой роли;
4. DTO privacy review;
5. повторный Supabase security advisor;
6. rollback rehearsal;
7. отдельный deploy PR с минимальными grants;
8. только после этого — официальный пункт меню и подключение frontend к RPC.

## Rollback

Поскольку prototype не применён, production rollback не требуется.

Для будущей изолированной среды порядок отката:

1. revoke будущих grants;
2. drop public consultation RPCs;
3. drop private consultation helpers;
4. drop `nav_consultation_events_v2`;
5. drop `nav_consultation_messages_v2`;
6. drop `nav_consultations_v2`.

Удаление production-данных без отдельного решения владельца запрещено.
