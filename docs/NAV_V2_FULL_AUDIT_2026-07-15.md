## Границы аудита

Проверены продуктовая логика, роли, UX, редкие сценарии, frontend, Supabase, RLS/RPC, Advisors, эксплуатационные данные, CI и release-процесс. Аудит выполнен read-only: production-данные, Auth, RPC и миграции не изменялись.

Не выполнялись пользовательские интервью и полноценная authenticated role/mutation matrix: соответствующий CI job сейчас пропускается. Deployed-код `nav-v2-deal-api` connector получить не смог.

## Итоговый вывод

Navigator v2 уже является сильным специализированным рабочим прототипом, а не простой анкетой. Хорошо реализованы:

- role-aware рабочие пространства;
- action-first интерфейс;
- адаптивный маршрут СПН;
- контроль рисков, документов, задач и ответственности;
- клавиатурная и screen-reader доступность;
- RLS и серверные проверки;
- широкие static/browser-регрессии.

Главный разрыв находится не в количестве предусмотренных сценариев. Сценариев уже достаточно много. Главный разрыв — между созданием контрольных пунктов и их фактическим завершением.

Текущая система хорошо выполняет цепочку:

`обнаружить пробел → создать задачу/документ/риск → показать приоритет`

Но пока не доказана цепочка:

`назначить владельца → выполнить → подтвердить результат → закрыть пункт → очистить очередь → продвинуть сделку`

## Сильные стороны

### Правильная граница продукта

Navigator помогает подготовить сделку: определить пробелы, документы, риски, ответственных и следующий шаг. Он не должен превращаться в полную CRM, телефонию, маркетинг и бухгалтерию.

### Ролевой интерфейс

СПН, менеджер, юрист, брокер, owner/admin и viewer получают разные рабочие акценты. Это снижает перегрузку универсального интерфейса.

### Top-3 и одно главное действие

Выделение трёх объяснимых приоритетов и одного главного действия в карточке — один из лучших элементов проекта. Его нужно сохранить как основной принцип.

### Адаптивный маршрут СПН

Маршрут зависит от задачи, стороны сопровождения, стадии, объекта, продавца, покупателя, денег и рисков. Большинство редких блоков не показываются без соответствующего триггера.

### Широкая предметная модель

Учтены квартиры, комнаты, доли, дома, земля, новостройки, коммерция, ипотека, маткапитал, НИС, сертификаты, несовершеннолетние, супруги, доверенности, обременения, банкротство, наследство, приватизация, перепланировка, альтернативные сделки и сложные расчёты.

### Доступность и feedback

Последние PR улучшили headings, groups, labels, field errors, native fieldset, клавиатуру, focus recovery, controlled dialogs, async feedback и mobile disclosure.

## Критические проблемы

### P0. Backlog не закрывается

Ни одной выполненной задачи и ни одного закрытого риска. Пока этот показатель не изменится на реальном пилоте, расширять каталог рисков и автоматических задач не следует.

Для каждого автоматического пункта должны быть:

- владелец;
- срок;
- критерий готовности;
- одно простое действие закрытия;
- подтверждённое сервером событие;
- связь с последующим статусом сделки.

### P0. Ответственность не соответствует процессу

В production нет активного профиля `manager`, но 18 сделок не имеют менеджера. Один активный СПН остаётся без manager_id. Нужны явные решения owner/admin по менеджерам, замещению, передаче и назначению юриста/брокера.

### P0. Все риски стали блокерами

Если каждый риск блокирует сделку, красный сигнал перестаёт быть полезным. Нужны уровни:

1. warning — обратить внимание;
2. gate — требуется подтверждение перед конкретным переходом;
3. stop — движение запрещено до решения профильной роли.

### P1. Нет `not applicable / waived / replaced`

Документ или проверка могут быть не нужны, заменены, объективно недоступны или ожидаться от внешней стороны. Без таких состояний backlog остаётся открытым навсегда.

### P1. Таксономия задач не встроена

Все `task_type` пусты. Без типа нельзя устойчиво применять SLA, дедуплицировать, строить аналитику и автоматически закрывать связанный риск или документ.

### P1. Не доказан командный цикл

Комментарии и review-записи отсутствуют. UI и RPC существуют, но реальное использование юристом и менеджером не подтверждено.

## UX и редкие сценарии

Основной подход правильный:

`общий триггер → короткое уточнение → специализированный mini-flow → передача профильной роли`

Редкий сценарий не должен показывать СПН полный юридический опросник. СПН фиксирует факты, система маршрутизирует, профильный специалист принимает решение.

Целевой принцип:

- редкий блок скрыт без триггера;
- не более 3–5 уточнений для СПН;
- вывод не маскируется под автоматическую юридическую истину;
- есть владелец, evidence и критерий закрытия;
- после закрытия пункт исчезает из основной очереди.

Необходимо разделить:

- факт: «собственник несовершеннолетний»;
- маршрутизацию: «передать юристу до задатка»;
- юридический вывод: решение юриста с датой, основанием и версией правила.

## Что не делать сейчас

- не добавлять ещё десятки юридических флагов;
- не создавать автозадачи без критерия закрытия;
- не считать все риски блокирующими;
- не удалять точные дубли без owner decision;
- не сужать grants без authenticated mutation tests;
- не переносить всю CRM в Navigator;
- не строить telemetry collector до privacy policy;
- не создавать отдельную страницу на каждый редкий случай;
- не считать green public CI подтверждением ролевой безопасности.

## Финальный вердикт

У проекта сильная продуктовая основа и высокий уровень инженерной дисциплины для рабочего прототипа. Однако его пока нельзя считать полностью готовой production-системой сопровождения сделок по четырём причинам:

1. не подтверждён замкнутый operational cycle;
2. не выполнена authenticated role/mutation matrix;
3. не решён критический контур персональных данных;
4. не реализован отдельный ПОД/ФТ-контур при его применимости.

Следующий этап:

`узаконить обработку данных → назначить ответственность → провести пилот → доказать закрытие backlog → упростить архитектуру → только потом расширять сценарии`.
