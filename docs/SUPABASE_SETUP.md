# Настройка Supabase для Навигатора сделок v2

Supabase является рабочей базой Навигатора сделок v2: через него работают вход, роли, сделки, документы, риски, задачи, комментарии, события и административные доступы.

- URL: `https://ofewxuqfjhamgerwzull.supabase.co`.
- Публичный сайт: `https://deputat36.github.io/vktg/`.

## Ключи

В браузерном коде можно использовать только publishable / anon key.

Файл подключения:

```text
config/supabase.js
```

Пример:

```js
export const SUPABASE_URL = 'https://xxxx.supabase.co';
export const SUPABASE_PUBLISHABLE_KEY = 'sb_publishable_...';
```

Нельзя хранить в репозитории:

- `service_role` key;
- secret key;
- любые приватные ключи и токены.

## Граница данных

Навигатор может использовать общий Supabase Auth, но прикладные данные должны оставаться только в зоне `nav_*` / `nav_v2_*`.

Основные таблицы v2:

- `nav_user_profiles`;
- `nav_deals_v2`;
- `nav_deal_participants_v2`;
- `nav_deal_documents_v2`;
- `nav_deal_risks_v2`;
- `nav_deal_tasks_v2`;
- `nav_deal_comments_v2`;
- `nav_deal_events_v2`;
- `nav_deal_reviews_v2`.

Запрещено использовать для Навигатора таблицы CRM «Лидер»: `leader_*`.

## Пользователи и роли

Пользователь входит через Supabase Auth. Его роль Навигатора хранится отдельно в `public.nav_user_profiles`.

Роли:

- `owner`;
- `admin`;
- `manager`;
- `spn`;
- `lawyer`;
- `broker`;
- `viewer`.

Административное создание доступа выполняется через Edge Function `nav-invite-user`. Функция должна требовать JWT и проверять, что текущий пользователь имеет роль `owner` или `admin`.

## Миграции

Все изменения базы оформляются SQL-файлами в:

```text
supabase/migrations
```

После изменения прав, таблиц, функций или политик нужно:

1. применить изменение в Supabase;
2. добавить миграцию в репозиторий;
3. проверить фактические права SQL-запросом;
4. запустить Supabase advisors.

## Безопасность

Обязательные правила:

- RLS должен быть включен на таблицах `nav_*` / `nav_v2_*`.
- Рабочие RPC не должны быть доступны `anon`, если они не предназначены для публичного сайта.
- `SECURITY DEFINER` функции должны сами проверять `auth.uid()` и роль пользователя.
- Изменения статусов, документов, задач и комментариев должны проходить через RPC, чтобы сохранялась история действий.
- Положительные статусы сделки должны блокироваться сервером, если есть незакрытые обязательные документы, проблемные документы или просрочка.


Она закрывает публичный вызов рабочих RPC:

- `nav_v2_get_lawyer_queue(integer)`;
- `nav_v2_return_spn_rework(uuid, text)`;
- `nav_v2_submit_spn_rework(uuid, text)`.

После применения проверено: `anon_can_execute = false`, `authenticated_can_execute = true`.

## Что проверить вручную после изменений

1. СПН входит и открывает `spn-v2.html`.
2. СПН создает тестовую сделку или открывает существующую.
3. Юрист открывает `queue-v2.html`.
4. Юрист меняет статус документа и оставляет комментарий.
5. Нельзя поставить положительный статус сделки при проблемных или незакрытых обязательных документах.
6. `nav-system-check-v2.html` не показывает критичных ошибок для текущей роли.
