# Navigator v2 — замыкание lifecycle задач

Дата фиксации: 24 июля 2026 года.

## Результат анализа

Read-only проверка production показала, что проблема находится не в отсутствии задач, сроков или ответственных:

- всего задач: 98;
- 88 задач остаются открытыми;
- 10 задач отменены;
- нет ни одной задачи `in_progress`;
- нет ни одной задачи `done`;
- все 98 задач имеют уже прошедший явный срок;
- среди задач, назначенных конкретным сотрудникам, нет отсутствующих или отключённых профилей;
- в истории сделок нет ни одного события `task_status_changed`.

Значит, существующий пользовательский сценарий не замыкает путь:

`открыта → в работе → результат → готово`.

Legacy RPC изменения статуса существует, разрешён роли `authenticated`, проверяет доступ к сделке и назначение задачи. Однако карточка показывала три равнозначные кнопки статуса одновременно и позволяла завершить задачу без отдельной фиксации результата.

## Что изменено

Для legacy-задач карточка теперь показывает только действие текущего этапа.

### Открытая задача

Показывается действие `Начать работу`.

После успешного сохранения задача должна перейти в статус `in_progress`, а карточка перезагрузиться с серверным состоянием.

### Задача в работе

Показываются:

- обязательное поле `Результат выполнения`;
- пояснение, какой результат нужно указать;
- действие `Сохранить результат и завершить`.

Результат сохраняется раньше статуса:

1. проверяется, что результат заполнен и содержит не менее 10 символов;
2. создаётся командный комментарий через действующий `nav_v2_add_comment`;
3. только после успешного комментария вызывается действующий `nav_v2_update_task_status(..., done)`;
4. карточка перезагружается и подтверждает серверное состояние через существующую историю сделки.

### Завершённая задача

Показывается только действие `Вернуть в работу`.

Оно предназначено для ситуации, когда ранее зафиксированный результат оказался неполным или устарел.

### Отменённая задача

Рабочие действия не показываются. При новой потребности должна создаваться новая актуальная задача.

## Частичный сбой

Frontend учитывает неатомарность двух существующих RPC.

Если комментарий уже сохранён, но смена статуса завершилась ошибкой:

- задача остаётся в текущем статусе;
- пользователь получает точное сообщение;
- повторная попытка в той же сессии не создаёт второй комментарий;
- повторяется только смена статуса.

Если комментарий сохранить не удалось, статус `done` не отправляется.

## Проверки

Добавлены и обновлены:

- pure model фаз задачи и результата;
- unit regression;
- desktop и mobile Playwright-сценарии;
- проверка обязательного результата;
- порядок `comment → done`;
- отказ комментария;
- отказ статуса после сохранённого комментария;
- повтор без дублирования комментария;
- права ответственного;
- возврат готовой задачи в работу;
- сохранение выключенного bounded transport;
- source/privacy/boundary checker.

## Заблокировано

### Атомарное завершение на сервере

Атомарное завершение на сервере, при котором результат и статус записываются одной транзакцией, заблокировано до отдельного разрешения на production database deployment.

Для него потребуется:

- новый или обновлённый RPC;
- migration и rollback package;
- проверка grants, RLS и actor scope;
- idempotency key;
- authenticated mutation E2E;
- отдельное подтверждение production применения.

Этот пункт не блокирует frontend-улучшение: текущий сценарий сохраняет результат раньше статуса и имеет безопасное восстановление частичного сбоя.

Bounded task transport также остаётся выключенным и не включался этим изменением.

## Граница изменений

- production Supabase не менялся;
- schema, data, RLS, Auth, grants и Edge Functions не менялись;
- существующие задачи массово не закрывались;
- `task_type` не заполнялся backfill-операцией;
- сотрудники не переназначались;
- `leader_*` не затрагивался.

## Следующий этап

После публикации следует контролировать появление первых событий `task_status_changed` и задач `in_progress`/`done` в scheduled read-only monitoring.

Если переходы начнут фиксироваться, следующий приоритет — аналогично проверить, что документы и риски получают подтверждённый terminal outcome и уходят из активного долга.

Если переходов не появится, следующий repository-only анализ должен проверить видимость вкладки задач, фактическое использование карточки сотрудниками и точность назначения роли, не создавая новый параллельный экран.
