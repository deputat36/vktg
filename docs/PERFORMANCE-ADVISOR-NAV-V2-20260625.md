# Performance Advisor: Навигатор v2

Дата первого анализа: 25 июня 2026.
Дата последней сверки: 29 июня 2026.

## Контекст

После исправления инцидента с карточкой сделки Овчинникова был запущен Supabase Performance Advisor.

Advisor вернул много предупреждений по разным частям общей базы, включая CRM «Лидер», старые таблицы `deals`, `orders`, `clients` и другие модули. Эти сущности не относятся к текущему проекту Навигатора v2 и не изменялись.

29 июня 2026 Performance Advisor был повторно просмотрен после синхронизации PR #154, расширения frontend RPC coverage и обновления cache-bust для карточки, списка сделок и админки.

## Что относится к Навигатору v2

Для текущего проекта релевантны предупреждения `multiple_permissive_policies` на таблицах:

- `public.nav_deal_answers_v2`
- `public.nav_deal_documents_v2`
- `public.nav_deal_expenses_v2`
- `public.nav_deal_participants_v2`
- `public.nav_deal_risks_v2`

Также advisor показывает часть `unused_index` предупреждений по таблицам `nav_*` и `nav_*_v2`. На текущем объеме данных это не является merge-блокером: индексы могли быть добавлены под будущие фильтры, роли и рост базы, а статистика использования в Supabase Advisor зависит от фактических запросов с момента сбора статистики.

## Причина предупреждения multiple permissive policies

На указанных таблицах одновременно есть две permissive-политики для роли `authenticated`, которые затрагивают SELECT:

- отдельная SELECT-политика, например `nav_v2_documents_select`;
- ALL-политика для записи, например `nav_v2_documents_write`.

Так как `ALL` включает SELECT, PostgreSQL вынужден учитывать несколько permissive-политик для одного и того же действия.

## Влияние на инцидент Овчинникова

Это не является непосредственной причиной ошибки карточки Овчинникова.

Причины:

- карточка загружается через `security definer` RPC;
- связанные таблицы карточки имеют индексы по `deal_id`;
- под JWT Овчинникова доступ к сделке подтверждён;
- lite-RPC и полная карточка на серверной стороне возвращают данные;
- frontend RPC coverage после PR #154 зеленый: `ok=true`, `items_count=38`, `problem_count=0`;
- RPC grants зеленые: `missing_authenticated_count=0`, `anon_open_count=0`, `public_open_count=0`.

## Почему RLS не исправлялось в PR #154

RLS-политики влияют на прямой доступ к таблицам, а не только на карточку. Механическое изменение `ALL`-политик на INSERT/UPDATE/DELETE может изменить поведение фронтенда или скрытых модулей, если где-то есть прямой доступ к таблицам.

Во время прохода 29 июня 2026 прямое чтение `pg_policies` через SQL-инструмент было заблокировано, поэтому определения live-политик не изменялись вслепую.

Исправление нужно делать отдельным этапом:

1. Получить определения текущих RLS-политик.
2. Проверить, какие страницы или модули обращаются к таблицам напрямую.
3. Заменить `ALL`-политики на отдельные политики записи без изменения логики SELECT.
4. Прогнать smoke-тесты:
   - карточка сделки;
   - безопасная карточка;
   - документы;
   - риски;
   - расходы;
   - участники;
   - мастер СПН;
   - список сделок для owner/admin/manager/lawyer/broker/spn.

## Что можно сделать отдельно после merge PR #154

1. Создать отдельную ветку для RLS performance cleanup.
2. Снять definitions всех `nav_v2_*` RLS policies.
3. Для write-политик проверить, какие из них созданы как `FOR ALL`.
4. Заменить `FOR ALL` на явные `FOR INSERT`, `FOR UPDATE`, `FOR DELETE`, где это безопасно.
5. Повторить Supabase Performance Advisor.
6. Если `unused_index` по `nav_*` сохранится после реальной нагрузки, анализировать только те индексы, которые не используются длительное время и не участвуют в планируемых фильтрах.

## Рекомендация

Не трогать RLS-политики в рамках PR #154. Вынести оптимизацию RLS и индексов в отдельную задачу после стабилизации интерфейса и ручной проверки по `docs/NAV_V2_MANUAL_QA.md`.

## CRM «Лидер» и другие модули

Advisor также показал много предупреждений по таблицам `leader_*`, `parket_*`, старым `deals`, `orders`, `clients` и другим сущностям. Они относятся к соседним или legacy-модулям. В рамках текущей работы Навигатор v2 не должен менять эти таблицы, их RLS или индексы.
