# Navigator v2 manual QA checklist

Короткий чеклист для ручной проверки PR #154 после merge/deploy.

## Список сделок

- Страница сделок открывается для рабочих ролей.
- В списке видно, кто ведет подготовку: СПН продавца, СПН покупателя или единый СПН сделки.
- Для роли СПН текст остается в формате зоны ответственности.
- Основной поиск находит сделки по СПН продавца, СПН покупателя, менеджеру, юристу и брокеру.
- Fallback поиска по СПН показывает прямые ссылки на найденные карточки только как страховка.

## Карточка сделки

- Карточка открывается без 400/401/timeout ошибок.
- При проблемах сессии появляется понятный блок повторного входа.
- Блок `СПН по сделке` показывает СПН продавца, СПН покупателя и менеджера.
- Если обе стороны ведет один СПН, строка объединяется.
- Значения ФИО и email не ломают верстку.
- Статусы документов, workflow документов, статусы задач и сроки задач сохраняются.
- Отправка СПН на повторную проверку после доработки работает.

## Админка команды

- Раздел доступен owner/admin и закрыт для остальных ролей.
- В карточке сотрудника понятно, где редактировать ФИО, телефон, роль, менеджера и активность.
- Кнопка сохранения подписана как `Сохранить данные`.
- СПН без менеджера получает визуальную подсказку.
- Активные профили без телефона получают визуальную подсказку.
- Уже выбранный выключенный менеджер подсвечивается как проблема.
- Назначение менеджера сохраняется.

## Создание доступа и вход

- Owner/admin может создать доступ новому СПН через `nav-invite-user` без ручного редактирования базы.
- При создании доступа с менеджером `manager_id` сохраняется в профиле сотрудника.
- Нельзя назначить менеджером выключенный профиль или роль, отличную от owner/admin/manager.
- Ссылка установки пароля открывает `nav-accept-invite-v2.html`.
- Страница принятия приглашения корректно обрабатывает recovery/invite токен и устанавливает новый пароль.
- После установки пароля пользователь попадает в рабочий интерфейс Навигатора.
- Неверный пароль на входе показывает понятное сообщение, без сырого технического текста.
- Повторный доступ для уже существующего пользователя не создает дубль профиля и обновляет данные сотрудника.

## Дизайн и удобство

- Главный экран, список сделок, карточка сделки, админка команды, мастер СПН и очередь юриста выглядят как единая система: одинаковые отступы, карточки, кнопки, статусы и заголовки.
- На мобильной ширине 360-430 px нет горизонтального скролла, обрезанных кнопок и наложения текста.
- На планшете и ноутбуке сетки не разваливаются: карточки читаются, кнопки остаются рядом с нужным действием.
- Цветовые статусы не конфликтуют: зеленый используется для успешного, желтый для предупреждения, красный для ошибки или срочного внимания, синий для информационных меток.
- Все важные действия визуально различимы: основное действие выделено сильнее, вторичные действия спокойнее, опасные действия отдельно.
- Длинные ФИО, email, адреса и комментарии переносятся и не ломают карточку.
- Пустые состояния выглядят осознанно: есть понятный текст, что данных нет или нужно сделать дальше.
- Ошибки и предупреждения написаны человеческим языком, без сырого технического текста там, где пользователь должен принять решение.
- Диагностические блоки owner/admin не перегружают обычные рабочие роли.
- После hard refresh не видны старые версии CSS/JS из кеша.
