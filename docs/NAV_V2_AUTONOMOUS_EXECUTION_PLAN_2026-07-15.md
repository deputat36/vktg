# Navigator v2 — автономный план доработки

Дата: 15 июля 2026 года  
Репозиторий: `deputat36/vktg`  
Основание: `docs/NAV_V2_FULL_AUDIT_2026-07-15.md`  
Стартовый `main`: `96ab9ba3921685a72d359440076a69af431b358f`

## Цель

Довести Navigator v2 от сильного специализированного прототипа до надёжного рабочего инструмента подготовки сделок, который:

- помогает СПН быстро и безопасно передавать сделку;
- не заменяет CRM и не заставляет дублировать лишние данные;
- показывает каждой роли только её текущую работу;
- не превращает редкие сценарии в постоянную перегрузку;
- замыкает задачи, документы и риски до подтверждённого результата;
- не допускает скрытых изменений production-данных;
- имеет доказанную ролевую безопасность;
- минимизирует юридические и compliance-риски.

## Главный продуктовый принцип

Каждый создаваемый системой контрольный пункт обязан иметь полный жизненный цикл:

`триггер → понятный пункт → ответственный → срок → действие → подтверждение → закрытие → влияние на сделку`

Нельзя расширять автоматическую генерацию backlog, пока не доказано его погашение.

## Порядок автономной работы

Работа выполняется короткими независимыми PR.

Для каждого PR:

1. Проверить актуальный `main`, открытые PR и связанные issues.
2. Выбрать один ограниченный сценарий.
3. Не смешивать UX, schema, security и data cleanup без необходимости.
4. Сохранить текущие роли, RPC и payload, если изменение контракта не является целью PR.
5. Добавить pure/static проверки и browser regression там, где меняется поведение.
6. Проверить desktop и mobile.
7. Проверить отсутствие утечки персональных данных в storage, URL, telemetry и artifacts.
8. Дождаться обязательных CI.
9. Проверить unresolved review threads.
10. Слить в `main`, если проверки зелёные и изменение не попадает под ручной gate.
11. Обновить `docs/NAV_V2_WORK_HANDOFF_LATEST.md` после законченной волны, а не после каждой микроправки.

## Что разрешено выполнять автономно

Без дополнительного согласования допускаются:

- исправления frontend-ошибок;
- улучшения доступности;
- улучшения busy/success/error feedback;
- безопасные UX-упрощения без изменения бизнес-правил;
- pure models и declarative policies;
- локальная консолидация модулей без смены поведения;
- новые тесты и CI-проверки;
- read-only диагностика GitHub и Supabase;
- документация и актуализация handoff;
- подготовка migration/RPC design в ветке без применения к production;
- небольшие additive изменения кода, не меняющие production schema/data/grants;
- merge таких PR после зелёного CI и отсутствия review threads.

## Что требует отдельного согласования

Нельзя выполнять автоматически:

- удаление, объединение или массовое закрытие production-строк;
- исправление дублей issue #273;
- назначение или замена manager/SPN/lawyer/broker в рабочих данных;
- изменение ролей и доступов реальных пользователей;
- создание платной Supabase preview branch;
- создание технических Auth users и secrets;
- применение production migrations, меняющих данные, grants, RLS или callable RPC surface;
- включение новых внешних сервисов, collectors и telemetry;
- перенос или копирование персональных данных;
- архивирование legacy Navigator;
- автоматический юридический вывод вместо решения профильного специалиста;
- действия, способные увеличить расходы без заранее установленного лимита.

## Приоритетная последовательность

### Волна 1. Замкнуть task-flow

Приоритет: P0/P1  
Начать немедленно.

Цель:

`нажать действие задачи → понять право → увидеть busy → получить server result → увидеть новый статус → при необходимости повторно открыть`

Работы:

1. Переиспользовать `page-action-feedback-v2.js` для task permission check и mutation.
2. Исключить ситуации, когда первая попытка только проверяет право и заставляет повторно нажимать без понятной причины.
3. Не скрывать задачу при permission error.
4. Показывать ответственную роль и понятный следующий шаг.
5. Проверить `open → in_progress → done → open`.
6. Проверить demo guard.
7. Сохранить `nav_v2_update_task_status` и текущий payload.
8. Не вводить новые task statuses.
9. Добавить desktop/mobile regression и source contract.
10. После PR выполнить read-only сверку production task closure.

Критерий завершения:

- действие доступно только разрешённой роли;
- busy/success/error доступны screen reader;
- повторное открытие работает;
- серверный результат отражается в карточке;
- RPC выполняется ровно один раз на подтверждённое действие;
- нет нового observer/storage/telemetry.

### Волна 2. Уменьшить вечный backlog без потери контроля

Приоритет: P0/P1.

Цель: подготовить нормальную модель закрытия документов, рисков и задач.

Подволны:

1. Провести inventory существующих статусов и переходов.
2. Спроектировать состояния:
   - `not_applicable`;
   - `waived`;
   - `replaced`;
   - `external_wait`;
   - `deferred`.
3. Для каждого состояния определить:
   - кто может установить;
   - обязательный комментарий/evidence;
   - влияет ли оно на readiness;
   - убирает ли пункт из основной очереди;
   - как отменяется решение.
4. Сначала сделать pure policy и fixtures.
5. Подготовить additive migration и RPC contract отдельным PR.
6. Production application оставить за отдельным gate.

Критерий завершения:

- ненужный пункт можно корректно снять с основной очереди;
- история решения сохраняется;
- состояние нельзя использовать для обхода stop-фактора без разрешённой роли;
- backlog metrics различают «в работе», «ожидание», «закрыто» и «не применимо».

### Волна 3. Встроить task taxonomy и критерии готовности

Приоритет: P1.

Проблема: все production-задачи имеют `task_type = NULL`.

Работы:

1. Составить bounded taxonomy по фактическим источникам задач.
2. Не делать сотни узких типов; начать с 8–12 устойчивых категорий.
3. Для типа определить:
   - owner role;
   - default SLA;
   - completion evidence;
   - связанный document/risk/deal gate;
   - правило дедупликации.
4. Обновить новые task creation paths.
5. Для старых строк сначала сделать read-only классификационный preview.
6. Не выполнять автоматический backfill без проверки выборки и approval.

Критерий завершения:

- новые задачи получают тип;
- одинаковые автоматические задачи не размножаются;
- менеджер видит backlog по типу и сроку;
- выполнение может закрывать связанный контрольный пункт по явному правилу.

### Волна 4. Провести реальный controlled pilot

Приоритет: P0, но требует участия владельца и сотрудников.

Пилот:

- 10–15 реальных сделок;
- минимум два СПН;
- менеджер;
- юрист;
- при наличии ипотечного сценария — брокер;
- длительность 10 рабочих дней.

Измерять:

- время создания заявки;
- число автоматически созданных пунктов;
- долю закрытых задач;
- долю закрытых/waived документов;
- долю разрешённых рисков;
- возвраты СПН;
- повторные вопросы юриста;
- среднее время до следующего действия;
- случаи, когда редкий сценарий мешал основному маршруту;
- случаи дублирования CRM.

Запрещено оценивать сотрудников по сырым метрикам пилота. Цель — проверить инструмент.

Критерий готовности к расширению:

- не менее 70% созданных контрольных пунктов получили корректный исход;
- нет скрытой потери данных;
- роли понимают своё главное действие;
- редкие сценарии не увеличивают обычный маршрут более чем на 3–5 уточнений;
- нет критичного обхода прав.

### Волна 5. Authenticated role/mutation matrix

Приоритет: P0 для release confidence.

Требует gate по стоимости issue #282.

После approval:

1. Создать изолированную preview branch с лимитом жизни.
2. Создать только технические аккаунты:
   - owner;
   - admin;
   - manager;
   - spn;
   - lawyer;
   - broker;
   - viewer.
3. Использовать синтетические данные.
4. Проверить desktop/mobile:
   - login/recovery;
   - dashboard;
   - deals;
   - card;
   - task/document/risk mutations;
   - SPN rework;
   - lawyer review;
   - viewer read-only;
   - owner/admin access;
   - forbidden cross-role operations.
5. Удалить branch/users/secrets после теста.

Критерий завершения:

- фактический PASS/FAIL по каждой роли;
- artifacts без PII;
- branch удалена;
- расходы уложились в утверждённый ceiling.

### Волна 6. Security hardening

Начинать после role/mutation matrix.

Работы:

1. Разделить публичный RPC surface на:
   - frontend API;
   - admin-only;
   - diagnostic;
   - internal helper;
   - legacy.
2. Убрать ненужные authenticated execute grants.
3. Проверить прямые table grants.
4. Удалить или изолировать legacy helpers только после решения issue #199.
5. Включить leaked-password protection после проверки влияния на onboarding/recovery.
6. Сузить CORS invite function до разрешённых origins.
7. Проверить deployed Edge source/version/hash.
8. Консолидировать security health checks.

Критерий завершения:

- anon surface = 0;
- authenticated surface совпадает с registry;
- admin/diagnostic функции не доступны обычным ролям;
- role matrix остаётся зелёной;
- Advisor warnings имеют documented disposition.

### Волна 7. Упростить frontend-архитектуру

Начинать после стабилизации процесса, не раньше.

Работы:

1. Инвентаризировать все SPN entry modules и DOM ownership.
2. Выделить единый draft store.
3. Ввести declarative step registry.
4. Перенести route/readiness/risk calculations в pure modules.
5. Убрать интервалы и observers там, где возможен explicit lifecycle.
6. Сократить повторные RPC загрузки.
7. Сохранить внешний UX и payload.
8. Делать по одному блоку, без полного rewrite.

Критерий завершения:

- один источник состояния;
- меньше order-dependent enhancements;
- сохранение и восстановление draft покрыты тестами;
- module budget снижается естественно, а не ради числа.

### Волна 8. Privacy и compliance foundation

Параллельно допустима только подготовительная работа без перемещения данных.

Работы:

1. Минимизировать PII в browser draft.
2. Добавить TTL, timestamps и явную очистку draft.
3. Очистить draft после successful save/logout.
4. Запретить PII в URL, telemetry и test artifacts.
5. Подготовить data inventory и retention matrix.
6. Определить legal owner каждого правила.
7. Создать versioned legal rule registry:
   - trigger facts;
   - source;
   - reviewed_at;
   - owner;
   - severity;
   - blocking scope;
   - required evidence;
   - resolution action.
8. Подготовить отдельный KYC/ПОД-ФТ mini-flow только после подтверждения профильным юристом.
9. Подготовить варианты data residency, не копируя production PII.

## Продуктовые правила для редких сценариев

Редкий сценарий допускается в основном UX только если:

- есть чёткий триггер;
- без триггера блок скрыт;
- СПН отвечает максимум на 3–5 фактических вопросов;
- система не выдаёт автоматический юридический вердикт;
- назначается профильная роль;
- есть evidence и критерий закрытия;
- после закрытия пункт исчезает из основной очереди;
- подробная юридическая часть раскрывается только профильной роли.

## Метрики успеха

Главные продуктовые метрики:

1. Доля задач с конечным исходом.
2. Доля рисков с разрешением или обоснованным сохранением.
3. Доля документов с конечным workflow state.
4. Среднее время от создания пункта до результата.
5. Доля пунктов без ответственного.
6. Доля просроченных пунктов.
7. Число повторных возвратов по одной причине.
8. Число автоматически созданных дублей.
9. Время СПН на заполнение типовой сделки.
10. Доля сделок с понятным следующим действием.

Нельзя считать успехом:

- число созданных задач;
- число экранов;
- число правил;
- число предупреждений;
- число CI workflows;
- снижение module budget без улучшения процесса.

## Decision gates

### Gate A — manager ownership

Нужно подтвердить:

- является ли Алексей Ковтун действующим manager для текущих СПН;
- должен ли owner оставаться отдельной ролью;
- кому назначать сделки без manager_id;
- правила замещения.

До решения выполняются только read-only previews и frontend reminders.

### Gate B — authenticated E2E budget

Рекомендация: разрешить одноразовую preview branch с общим ceiling до 0,10 USD и максимальным временем жизни 6 часов. Любое превышение или продление требует повторного согласования.

### Gate C — data residency

Рекомендуемое направление:

- не расширять состав PII в текущем зарубежном production;
- подготовить перенос primary storage персональных данных на инфраструктуру, соответствующую российским требованиям;
- Supabase оставлять только после юридически подтверждённой архитектуры либо использовать для обезличенных/технических данных.

До решения не выполнять перенос или копирование production PII.

### Gate D — 115-ФЗ

Нужно письменное заключение профильного специалиста о применимости и составе обязанностей конкретной организации. До него Navigator может фиксировать общие риск-факты, но не должен заявлять, что выполняет обязательный compliance-контур.

### Gate E — exact duplicates

Четыре группы issue #273 удаляются/объединяются только после ручного решения по каждой паре с выбором канонической карточки.

## Ближайшая автономная очередь

1. Task permission/action feedback и closure regression.
2. Read-only task lifecycle/data-quality сверка.
3. Pure policy для backlog terminal states.
4. Task taxonomy registry v1.
5. Browser draft TTL/cleanup без изменения server data.
6. CI consolidation только после завершения продуктовых P0.

## Условия остановки

Автономная работа останавливается и выносится вопрос владельцу, если:

- требуется production data mutation;
- требуется платный ресурс;
- есть несколько разумных бизнес-решений с разными последствиями;
- изменение может повлиять на юридическую ответственность;
- тесты показывают конфликт текущих ролей или данных;
- CI не даёт достоверного evidence;
- обнаружен возможный security incident;
- изменение может нарушить работу текущих пользователей.
