# CRM Навигатор сделок v2

Веб-приложение для подготовки и контроля сделок с недвижимостью в офисе. Проект помогает СПН, юристу, ипотечному брокеру, менеджеру и администратору быстрее собрать сделку, увидеть риски, не потерять документы и не перейти к задатку или основной сделке слишком рано.

## Главная цель

Навигатор должен заменить хаотичную переписку и ручные памятки понятным рабочим маршрутом:

1. СПН заполняет новую сделку через умный мастер.
2. Система определяет риски, документы, задачи, юриста и брокера.
3. Юрист видит отдельную очередь по стоп-факторам, документам и задаткам.
4. Менеджер контролирует зависшие сделки, доработки и красные риски.
5. Новичок учится прямо в процессе заполнения, потому что вопросы идут по логике сделки.

## Основные страницы

- `nav-v2.html` — вход и стартовая страница.
- `dashboard-v2.html` — рабочий стол пользователя.
- `spn-v2.html` — мастер создания новой сделки.
- `deals-v2.html` — список доступных сделок.
- `deal-card-v2.html` — карточка сделки.
- `queue-v2.html` — кабинет юриста.
- `admin-v2.html` — команда и управление доступами.
- `nav-system-check-v2.html` — диагностика входа, роли и Supabase.
- `spn-v2-checklist.html` — ручная проверка сценариев мастера СПН.

## Роли

- `spn` — создает и ведет свои сделки.
- `lawyer` — проверяет риски, документы, задатки и юридические решения.
- `broker` — подключается к ипотеке, сертификатам, маткапиталу и банковским вопросам.
- `manager` — контролирует сделки команды.
- `admin` / `owner` — управляет пользователями, доступами и диагностикой.
- `viewer` — ограниченный просмотр.

## Supabase

Рабочий проект Supabase: `ofewxuqfjhamgerwzull`.

В браузерном коде разрешен только publishable / anon key. Нельзя добавлять в репозиторий secret key или service_role key.

Данные Навигатора должны храниться только в зоне:

- таблицы `nav_*` и `nav_v2_*`;
- функции `nav_*` и `nav_v2_*`;
- профили `nav_user_profiles`.

Нельзя использовать таблицы CRM «Лидер» для данных Навигатора: `leader_clients`, `leader_orders`, `leader_user_profiles` и другие `leader_*`.

## Что важно при доработке

- Все изменения базы оформлять миграциями в `supabase/migrations`.
- Проверять RLS и права RPC после каждого изменения Supabase.
- Важные запреты должны жить на сервере, а интерфейс должен объяснять пользователю причину.
- Не перегружать СПН техническими экранами: рабочий маршрут должен быть простым.
- Не ломать роли СПН, юриста, брокера, менеджера и администратора.
